# Release 0.27.0 — Fase 16d · Telas de operação: People/RH, Ponto, Assets, Admin, Platform/TI e Identity/acesso (fecha a Fase 16)

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 16 (fatia 16d — **última**)
- **Specs entregues:** **SPEC-0029** (estendida — 16d implementa People/Ponto/Assets/Admin/Platform/Identity; **Fase 16 concluída**)
- **Decisão de origem:** DL-0109 (gap de UI: telas de módulo adiadas nunca construídas)
- **Tag git:** `0.27.0`

## Destaques

A Fase 16d **quita o restante da dívida de UI** (DL-0109), agora no **back-office e na TI**. É a
**última das quatro fatias da Fase 16**: com ela, **todo módulo que só existia como API ganha tela de
operação** — o operador enxerga o ERP inteiro. Entrega **seis telas de operação**, **frontend-only,
sobre APIs que já existiam** (nenhum endpoint novo, nenhum contrato ou schema tocado). Reusa o padrão
estabelecido: service + `API_BASE_URL` + `PageResponse`; `<app-screen-state>` para carregando/vazio/
erro/permissão; rota lazy sob o `Shell` com guards; nav por papel (só para esconder ruído de menu);
i18n bilíngue; Vitest + Playwright.

## Adicionado

- **Tela People/RH** (`/people`): **colaboradores** (listar com filtro de situação + registrar);
  **jornada e banco de horas** de um colaborador/período (horas trabalhadas × contratadas × saldo — tudo
  vindo do backend, sem aritmética no cliente); e a **fila de discrepâncias** (filtros período/situação —
  aviso, sem correção automática). Consome `GET/POST /api/people/employees`, `GET /employees/{id}/journey`,
  `GET /employees/{id}/timebank`, `GET /discrepancies`. O holerite continua sendo arquivado no cofre pelo
  fluxo multipart existente (fora da tela).
- **Tela Ponto** (`/point`): **histórico de coletas** do REP (filtro de situação, com tentativas/itens/
  classe de falha) e **espelho operacional** por id — leituras de operador/TI. Consome `GET
  /api/integration/point/runs` e `GET /snapshots/{id}`. A ingestão do AFD/AEJ assinado (`POST /afd`) e o
  disparo de coleta (`POST /crawl`) são **contratos máquina-a-máquina/operacionais** e **não ganham tela**.
- **Tela Assets/Patrimônio** (`/assets`): **listar** com filtros combináveis type/status/expiringWithinDays,
  **registrar** (custo na moeda original), **baixar** com motivo auditado e **varredura de licenças a
  vencer** (mostra o total sinalizado). Consome `GET/POST /api/assets`, `GET /{id}`, `POST /{id}/retire`,
  `POST /flag-expiring`.
- **Tela Admin/Back-office** (`/admin`): **fornecedores administrativos** (listar/registrar), **contratos**
  por fornecedor (listar/registrar), **despesa recorrente** (registrar — cria o lançamento no Financeiro e
  lista os documentos exigidos) e **varredura de contratos a vencer**. Consome `GET/POST
  /api/admin/suppliers`, `GET/POST /suppliers/{id}/contracts`, `POST /expenses`, `POST
  /contracts/flag-expiring`. As **escritas exigem `ROLE_FINANCE`** (DL-0088) — sem o papel, 403 renderizado
  como estado de permissão.
- **Tela Platform/TI** (`/platform`): **catálogo de jobs governados** + **histórico de execuções**
  (filtros job/status) + **disparo manual** (ROLE_IT; 404 desconhecido, 409 já rodando); **status do
  certificado e-CNPJ** — **só metadados** (titular/documento/impressão digital/validade/dias/situação); a
  **chave e a senha NUNCA trafegam à UI**; e a **auditoria de sistema** consolidada (filtros actor/type/
  janela). Consome `GET /api/platform/jobs`, `GET /jobs/runs`, `POST /jobs/{name}/trigger`, `GET
  /certificate/status`, `GET /audit`.
- **Tela Identity/acesso** (`/identity`): **catálogo de papéis/permissões** (fonte da verdade da
  autorização interna) e **auditoria de acesso** (login/negação, filtros actor/type/janela). Ambas exigem
  **DIRECTOR ou IT** — sem o papel, 403 (estado de permissão). Consome `GET /api/identity/roles` e `GET
  /access-audit`. O login é no IdP OIDC externo (Fase 13); não há gestão de credencial na tela.
- **Navegação e i18n:** seis itens de nav novos — People/Ponto/Assets/Platform `roles: ['ROLE_IT']`
  (**não existe papel "HR"** no realm), Admin `['ROLE_FINANCE']`, Identity `['ROLE_DIRECTOR','ROLE_IT']`;
  blocos i18n pt-BR e en-US por tela. As telas cuja inicial colide com uma rota anterior seguem acessíveis
  pela paleta `Ctrl/Cmd+K`.
- **Testes:** Vitest por tela + specs de serviço (`HttpTestingController`) → **264 testes de frontend**
  (49 arquivos), cobertura acima dos pisos da Fase 12; jornada Playwright `e2e/platform-people.spec.ts`
  (IT abre Platform → certificado só-metadados + catálogo de jobs, depois People → lista vazia; token sem
  ROLE_IT recebe 403 no trigger de job governado, o IT é autorizado — autoridade no backend).

## Alterado

- **`backend/pom.xml` + OpenAPI**: versão `0.26.0 → 0.27.0` (apenas a string de versão e o texto da
  descrição do OpenAPI, que passa a registrar 16c/16d como frontend-only e a Fase 16 concluída).
  `./mvnw verify` permanece **verde — 476 testes, BUILD SUCCESS** (nenhuma mudança de comportamento no
  backend).

## Não alterado (frontend-only)

- **Nenhum endpoint novo**, nenhuma migração, nenhum contrato/JSON/evento/schema mudado. As telas
  consomem os endpoints existentes.

## Contratos / API

- **Sem mudança.** As telas consomem `/api/people`, `/api/integration/point`, `/api/assets`, `/api/admin`,
  `/api/platform` e `/api/identity` já existentes. A autorização é do backend: as leituras de People/Assets
  e Ponto caem em `authenticated()`/`permitAll` (Ponto está sob o ACL M2M por HMAC); **Platform
  jobs/trigger e certificate exigem `ROLE_IT`**, **admin POST exige `ROLE_FINANCE`**, **identity
  roles/access-audit exigem `ROLE_DIRECTOR`/`ROLE_IT`** — a nav só reflete isso como tidiness. O certificado
  expõe **só metadados** (nunca a chave/senha).

## Como rodar / verificar

```bash
cd frontend && npm ci && npx ng lint && CI=true npx ng test --watch=false && npx ng build
# lint verde; 264 testes Vitest verdes; cobertura acima dos pisos (stmts 73.4% / branches 56.5% /
# funcs 51.1% / lines 78.8%); build gera os chunks people/point/assets/admin/platform/identity-page.

cd backend && ./mvnw verify   # 476 testes, BUILD SUCCESS (inalterado)

# E2E (stack isolado, Keycloak na 8089):
cd frontend && npm run e2e:up && E2E_BASE_URL=http://localhost:4201 npm run e2e && npm run e2e:down
# 19 specs Playwright (17 anteriores + a jornada Platform/People de 16d).
```

## Riscos e pendências

- **E2E no sandbox:** a jornada Playwright foi **autorada e compila** (o Playwright a descobre — 19 testes
  via `npx playwright test --list`), mas **não foi executada aqui** porque a build da imagem do backend do
  stack isolado (`compose.e2e.yaml`) exige rede/cache Maven dentro do contêiner, indisponível no sandbox.
  O `./mvnw verify` **no host** está verde, então é limitação de infra, não defeito de código.
- **Fase 16 concluída:** não há mais fatias de UI de operação pendentes — toda a dívida de UI (DL-0109)
  está quitada.
