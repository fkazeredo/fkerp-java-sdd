# Release 0.28.0 — Fase 17 · Remover Keycloak → Authorization Server self-hosted (Spring Authorization Server embutido)

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 17
- **Specs entregues:** **SPEC-0024** (re-graduada — OIDC self-hosted; a graduação da Fase 13 vira histórico)
- **Decisões:** ADR-0018 + DL-0110, DL-0111, DL-0112, DL-0113, DL-0114
- **Tag git:** `0.28.0`

## ⚠️ Breaking change (permitido em `0.y`, ADR 0015 §4)

**O Keycloak foi removido 100%.** O IdP OIDC deixa de ser um contêiner externo e passa a ser servido
pelo **próprio app** (Spring Authorization Server embutido). Consequências para quem opera:

- O serviço `keycloak` saiu do `docker-compose.yml` e do `compose.e2e.yaml`; a pasta `infra/keycloak/`
  (realm export + README) foi apagada; as variáveis `KEYCLOAK_*` saíram do `.env.example`.
- A URL do OIDC (`OIDC_ISSUER_URI`) agora aponta para o **próprio app** (`http://localhost:8080` em dev,
  `http://localhost:8081` no E2E), não mais para um realm do Keycloak.
- **Nenhum contrato `/api` mudou** — o esquema de segurança OpenAPI passou de "OIDC do Keycloak" para
  "OIDC self-hosted (AS embutido)". O `POST /api/identity/login` in-house **continua removido** (foi
  aposentado na Fase 13).

## Destaques

A Fase 17 atende à decisão do dono de **não usar Keycloak**. Trocamos **apenas o IdP**: o app agora é
**IdP e Resource Server ao mesmo tempo**, servindo OIDC pelo **Spring Authorization Server embutido**
(sem novo processo/Docker — recomendação do ROADMAP). O truque que tornou a troca mínima: o access
token continua carregando o **mesmo claim de papéis `realm_access.roles`** que o Keycloak emitia, então
o Resource Server, o mapeamento de papéis, a porta `UserContextProvider` e os **480 testes de backend**
ficaram **inalterados** — só a *origem* do token mudou (o próprio app).

## Adicionado

- **Authorization Server embutido** (`infra/security/AuthorizationServerConfig`): três
  `SecurityFilterChain`s — a do AS (`@Order(1)`), a do Resource Server `/api/**` (`@Order(2)`) e a de
  form-login `/login` (`@Order(3)`). Expõe `/.well-known/openid-configuration`, `/oauth2/authorize`,
  `/oauth2/token`, `/oauth2/jwks`, `/userinfo` e o form `/login`. Assina RS256 com uma chave RSA local
  (JWKS servido pelo próprio AS). Um `OAuth2TokenCustomizer` injeta `realm_access.roles` +
  `preferred_username` no access token (DL-0110).
- **Client SPA público** `acme-erp-web` registrado no AS (Authorization Code + PKCE S256, sem secret,
  sem tela de consentimento; redirects para 4200/4201). Espelha o antigo client do realm (DL-0111).
- **Store local de usuários reintroduzido** (migração **V32**, idempotente): recria `identity_users` +
  `user_roles` (BCrypt), com um `UserDetailsService` e um **seeder dev/E2E** (`dev` + um por papel, senha
  `dev12345`, **apenas** em dev/E2E). O catálogo papel→permissão (`roles`/`role_permissions`) permanece
  intacto (DL-0112).
- **CORS** nos endpoints do AS para a SPA (origens 4200/4201) buscar discovery/JWKS/token cross-origin.

## Alterado

- **Resource Server reapontado**: `issuer-uri`/`jwk-set-uri` apontam para o próprio app; o
  `SecurityConfig` ganhou `@Order(2)` e um `securityMatcher` para o `/api/**`+actuator+docs (o login e o
  fluxo OAuth do browser ficam com as cadeias do AS/form-login). Comportamento de autorização
  **inalterado**.
- **Frontend** (`core/auth/oidc.config.ts`): o `issuer` passa a ser o próprio app; o silent-refresh
  passa a ser por **iframe silencioso** (o SAS não emite refresh token a client público — DL-0113). O
  `angular-oauth2-oidc`, o interceptor, o guard e a API pública do `AuthService` **não mudaram**. Novo
  `public/silent-refresh.html`.
- **E2E** re-autorados contra o AS self-hosted: o login usa o form `/login` do próprio app; a obtenção
  de token para checagem de papel usa o fluxo real code+PKCE no browser (o SAS, OAuth 2.1, não tem
  direct-grant). As 19 jornadas compilam (`playwright test --list`).
- **OpenAPI** e Javadoc de auth atualizados (esquema de segurança → OIDC self-hosted; versão 0.28.0).

## Removido

- O serviço `keycloak` do `docker-compose.yml` e do `compose.e2e.yaml`; `infra/keycloak/`
  (`realm-acme.json` + `README.md`); as variáveis `KEYCLOAK_*` do `.env.example` (DL-0114).

## Migrações

- **V32__reintroduce_local_user_store.sql** — recria `identity_users` + `user_roles` (idempotente,
  `CREATE TABLE IF NOT EXISTS`; nunca edita a V29/V31 aplicadas). Mantém `roles`/`role_permissions`.

## Impacto em OpenAPI / contratos

- Nenhum endpoint `/api` novo ou alterado. O **esquema de segurança** passou a documentar o **OIDC
  self-hosted** (AS embutido, JWKS do próprio app). Versão do contrato → **0.28.0**.

## Riscos e o que fica para depois

- **Chave RSA de assinatura em memória** (gerada no boot; app single-instance — ADR 0002): tokens são
  invalidados a cada restart. Produção multi-instância exigiria chave persistida/externalizada (seam
  registrado; fora do escopo — Regra Zero).
- **Usuários voltam a viver no ERP** (store BCrypt) — o ERP volta a custodiar hash de senha; mitigado por
  hash-only + seed forte só em dev/E2E; o seam `UserDetailsService` permite plugar um diretório depois.
- **Silent-refresh por iframe** depende da sessão SSO do AS (mesma origem) — comportamento padrão do
  `angular-oauth2-oidc` para SPA sem refresh token.
