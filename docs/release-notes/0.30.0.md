# Release 0.30.0 — Fase 18b · enums de referência → cadastros (Marketing/Intelligence/Portfolio) + rótulos nas telas

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 18 (fatia 18b de 4)
- **Specs entregues:** **SPEC-0031** (atualizada — tipos 18b marcados como entregues)
- **Decisões:** **ADR-0019** (padrão enum→cadastro) + **DL-0116** (enums Marketing/Intelligence/Portfolio)
- **Tag git:** `0.30.0`

## Destaques

Segunda fatia da Fase 18: reusa o módulo `cadastro` (18a) e converte mais **três grupos** de enums de
referência para **`code` validado**, mantendo o contrato de fio inalterado (**`code` = nome do antigo
constante do enum ⇒ JSON idêntico**). Além disso, **retro-corrige o seam de 18a**: as telas passam a
mostrar o **rótulo** do cadastro em vez do código.

- **Marketing:** `ConsentPurpose`, `SubjectType` → `code` validado (`CONSENT_PURPOSE`,
  `MARKETING_SUBJECT_TYPE`), validados na escrita (`grantConsent`).
- **Intelligence:** `SubjectKind`, `InsightType`, `Verdict` → `code` (`INSIGHT_SUBJECT_KIND`,
  `INSIGHT_TYPE`, `INSIGHT_VERDICT`). São **produzidos pelo sistema** (o DSS os cunha de eventos
  consumidos), então viram cadastro para rótulo/label — sem validação de escrita.
- **Portfolio:** `GoalMetric` → `code` (`GOAL_METRIC`), validado no `defineGoal`. A ramificação da
  projeção realizada é preservada: **VOLUME←BookingConfirmed, REVENUE←SpreadRealized** (DL-0062).

## Adicionado

- **Constantes `*Codes`** (comportamento cablado, DL-0116): `MarketingCodes` (NEWSLETTER + varredura
  de purposes na anonimização LGPD), `IntelligenceCodes` (AGENCY/PROMO_FX_ADVISOR/CONVERTE/
  QUEIMA_MARGEM — regra da guardrail e narrator), `GoalMetricCodes` (VOLUME/REVENUE — forma do alvo e
  projeção).
- **Lookup de rótulo no frontend:** `CadastroLabelService` (busca `GET /api/cadastro/items?type=…`,
  cacheia o mapa `code→label` por tipo) + `CadastroLabelPipe` (`{{ code | cadastroLabel:'TIPO' }}`),
  aplicado nas telas **Marketing / Inteligência / Portfólio**. i18n `marketing.purpose` (pt-BR + en).
- **Migração V34** semeia os valores atuais dos 6 tipos (13 itens; `code`=nome do enum, `label`
  pt-BR), idempotente (`ON CONFLICT DO NOTHING`).

## Alterado

- **Marketing/Intelligence/Portfolio:** campos de enum (`ConsentPurpose`, `SubjectType`,
  `SubjectKind`, `InsightType`, `Verdict`, `GoalMetric`) passaram a `String code` (entidades, DTOs,
  views, eventos, repositórios, serviços; params de controller `String`). Os enums foram removidos.
- **Telas Marketing/Inteligência/Portfólio** exibem o **rótulo** do cadastro (não o code) — corrige o
  seam que 18a deixou nas telas.
- **OpenAPI** (versão → 0.30.0): os campos convertidos mantêm o schema `string` — **nenhum contrato
  `/api` existente mudou**.

## Removido

- Os enums `ConsentPurpose`, `SubjectType` (marketing), `SubjectKind`, `InsightType`, `Verdict`
  (intelligence), `GoalMetric` (portfolio) — substituídos por `code` validado + constantes `*Codes`.

## Migrações

- **V34__seed_cadastro_marketing_intelligence_portfolio.sql** — semeia os 6 tipos 18b em
  `cadastro_item`. Idempotente; nunca edita uma migração aplicada.

## Impacto em OpenAPI / contratos

- Os campos convertidos continuam `string` no JSON — **nenhum contrato existente quebrou** (invariante
  DL-0116). Sem novos endpoints. Versão do contrato → **0.30.0**.

## Riscos e o que fica para as próximas fatias (18c–18d)

- **Métrica de meta nova (não cablada):** um `GOAL_METRIC` novo cai no ramo VOLUME-shaped e **não tem
  projeção realizada** até uma fatia futura cablar — seam documentado (DL-0116).
- **Veredito desconhecido** em Intelligence é tratado como inválido (`isValid` → `default false`),
  fallback sem persistir insight.
- **Sem trava de obrigatoriedade** (desativar o último item ativo de um tipo) — Open Question da
  SPEC-0031, adiada.
- **18c–18d** convertem os grupos restantes (Compliance/People/Booking/Payout/AfterSales…) reusando o
  módulo, uma fatia por grupo.
