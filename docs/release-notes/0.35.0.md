# Release 0.35.0 — Fase 19c · Hardening (segredos fail-fast, anti-replay, cofre, lockout)

- **Data:** 2026-07-02
- **Fase do ROADMAP:** 19 (fatia 19c de 12 — Refactoring de maturidade)
- **Specs entregues:** SPEC-0024 (BR4/lockout), SPEC-0009/0017 (webhooks), SPEC-0008 (cofre), SPEC-0016 (gate de emissão)
- **Decisões:** DL-0122 (anti-replay), DL-0123 (fail-fast de produção), DL-0124 (upload/cofre), DL-0125 (lockout + política de senha)
- **Tag git:** `0.35.0`

## Destaques

Terceira fatia da Fase 19: fecha os riscos de segurança de bastidor apontados pela auditoria.
Nada visível ao usuário final muda (o MANUAL não é tocado — Regra Zero); tudo é postura de
segurança/infra.

- **Anti-replay nos webhooks (DL-0122).** A assinatura HMAC passa a cobrir
  **`timestamp + "." + corpo`** com **janela de tolerância** (default 300s), num **helper único**
  (`WebhookSignatures`) que elimina a duplicação entre os 2 verificadores. Um corpo validamente
  assinado não pode mais ser **reenviado para sempre**. Header de timestamp exigido
  (`X-Signature-Timestamp` / `X-Payment-Signature-Timestamp`) — **breaking M2M** (os pares são
  nossos/emulados).
- **Fail-fast de produção (DL-0123).** `ProdReadinessValidator` (perfil `prod`) **recusa subir**
  com segredo default/vazio (webhooks, `PLATFORM_SECRET_KEY`, senha de DB `acme`), issuer
  `http://` ou `billing.tax.regime-confirmed=false` (gate de NFS-e real — DL-0121). Nenhum valor
  de segredo é logado.
- **Upload do cofre endurecido (DL-0124).** Validação por **magic bytes** dos binários
  (pdf/png/jpg) — fecha a divergência doc×código ("never trust the extension alone" era só
  Javadoc); `fileRef` validado como UUID em read/delete (defesa em profundidade contra traversal).
- **Lockout de login (DL-0125).** Força-bruta no form login do AS embutido: **5 falhas
  consecutivas → bloqueio de 15 min** (contador DB, V38), aplicado via `accountLocked` no
  `UserDetailsService`; erro genérico (BR4). **Política mínima de senha** (≥8, não-repetido).

## Impacto em OpenAPI / contratos

- Os 2 webhooks passam a **exigir** o header de timestamp (breaking M2M). Nenhum contrato de
  usuário `/api` muda de shape.

## Portões

- Backend `./mvnw verify` **552 testes** (528→552, +24) BUILD SUCCESS; ArchUnit/Modulith/Spotless/
  Checkstyle/JaCoCo verdes.
- Frontend inalterado (nenhuma mudança de tela).

## Riscos

- Um cliente M2M que não enviar o header de timestamp recebe 401 — **intencional**; os pares
  reais/emulados assinam com timestamp.
- O fail-fast só age em `prod`; dev/test/e2e mantêm defaults (por construção).
