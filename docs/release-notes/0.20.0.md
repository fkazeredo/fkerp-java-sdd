# Release 0.20.0 — Admin: fornecedores e contratos administrativos → Finance + Compliance

- **Data:** 2026-06-30
- **Fase do ROADMAP:** 8l (Admin — SPEC-0025, subdomínio **genérico**) — **última sub-fase do bloco 8x**
- **Specs entregues:** SPEC-0025 (cadastro de fornecedores administrativos e contratos; despesa
  recorrente que gera lançamento no Finance e referencia o documento no Compliance; alerta de contrato
  a vencer)
- **Tag git:** `0.20.0`

## Destaques

A Fase 8l entrega o **balcão administrativo**: o registro **enxuto** dos custos administrativos
recorrentes (luz, água, telefone, software/serviço PJ, autônomo) e dos contratos que os sustentam, de
modo que cada despesa gere o **lançamento certo** no Finance (SPEC-0015) e **referencie o documento
hábil** exigido pelo Compliance (SPEC-0008). É o **22º módulo Modulith** (`com.fksoft.domain.admin`,
DL-0084), um **genérico** entregue como **registro + seam** — **procurement completo**
(cotação/compra) fica fora de escopo (comprar se exigido). Três fatias:

1. **Fornecedores + contratos + gate de papel (8l-1, BR1/BR2/BR6, DL-0084/0088).** Cadastro de
   `AdminSupplier` (UTILITY | SOFTWARE | SERVICE | OTHER, nasce ACTIVE) e `AdminContract` (vigência,
   recorrência, valor, documento do Compliance **por valor** — nunca FK). Janela de contrato inválida →
   400. As **escritas** exigem **`ROLE_FINANCE`** (a despesa vira obrigação financeira): sem o papel →
   **403 auditado**; com o papel → 201. Toda alteração é **auditada** no `system_audit` do Platform
   (`ADMIN_CHANGE`, **metadados only** — o CNPJ/CPF nunca aparece inteiro, pode ser dado pessoal de
   autônomo).
2. **Despesa recorrente → Finance + documentos exigidos (8l-2, BR3/BR4, DL-0085/0086).** Registrar uma
   despesa **cria** um lançamento **PAYABLE** no Finance pela **fachada** `FinanceService.register`
   (sem FK), guarda o `financeEntryId` por valor e **aponta os documentos exigidos** pelo Compliance
   (porta de leitura `DocumentRequirementDirectory`). O `kind` mapeia o `entryType` e os documentos
   (DL-0085): **UTILITY**→`UTILITY_EXPENSE`/[UTILITY_BILL]; **autônomo**→`AUTONOMOUS_SERVICE`/[RPA];
   **PJ**→`SERVICE`/[NFSE]; **OTHER**→`OTHER_EXPENSE`/[]. É **idempotente** por
   `(fornecedor, período, tipo)` (duplicada → 409, sem segundo lançamento). Admin **não** impõe o
   documento nem fecha o período (BR4) — só **gera** o lançamento e **referencia** o documento; o
   veto/fechamento continua sendo Finance+Compliance. **A regra de ouro vale também para o
   administrativo:** uma conta de consumo **sem** a fatura **impede o mês de fechar** (a regressão prova
   `canClose=false` antes de anexar, `canClose=true` depois).
3. **Alerta de contrato a vencer (8l-3, BR5, DL-0087).** `AdminContractExpiring` é publicado por um
   **job de relógio controlado** (horizonte 30 dias, idempotente por contrato), governado pelo Platform
   (`GovernedJobs` + `AdminContractExpiryScheduler`, com lock/janela/histórico de execução). É
   **alerta — nunca bloqueia**. Endpoint ad-hoc `POST /api/admin/contracts/flag-expiring` para acionar
   a varredura.

## Regras de negócio (SPEC-0025)

- **BR1** fornecedor com type/identifier/status (ACTIVE/INACTIVE). **BR2** contrato com vigência,
  recorrência, valor e documento (Compliance, valor). **BR3** despesa cria `LedgerEntry` PAYABLE com o
  `entryType` certo e sinaliza os documentos exigidos. **BR4** Admin **não** impõe documento nem fecha
  período — só gera o lançamento e referencia o documento. **BR5** contrato a vencer publica
  `AdminContractExpiring` (alerta). **BR6** toda alteração auditada. **BR7..BR11** ASSUMIDO (ver
  DL-0084..0088).

## Eventos

- `AdminSupplierRegistered {supplierRef, occurredAt}` / `AdminContractRegistered {supplierRef,
  occurredAt}` — produtor `admin`.
- `AdminExpenseRegistered {expenseId, financeEntryId, entryType, occurredAt}` — consumidores `finance`/
  `compliance`/`intelligence` (rastreio; o lançamento em si é criado de forma síncrona pela fachada).
- `AdminContractExpiring {contractId, validUntil, occurredAt}` — alerta (não bloqueia).

## Migração

- **V30__create_admin.sql** (idempotente): `admin_suppliers`, `admin_contracts` (+
  `expiry_signaled_at`), `admin_expenses` (UNIQUE `(supplier_id, period, kind)`); FKs **internas** ao
  módulo (contrato/despesa → fornecedor); referências a Finance/Compliance **por valor** (uuid), sem FK
  cross-contexto. **Seed aditivo** do catálogo do Compliance para o `SERVICE`
  (`NFSE` AT_REGISTRATION; `PAYMENT_PROOF` AT_SETTLEMENT) **sem editar a V8 já aplicada**. **Seed** do
  job `admin-contract-expiry` no catálogo do Platform (`scheduled_jobs`).
- **Finance `EntryType`** ganhou `SERVICE` e `OTHER_EXPENSE` (aditivo — não muda os tipos existentes).

## Contratos / OpenAPI

- Novos endpoints `/api/admin`: `POST /suppliers`, `GET /suppliers/{id}`,
  `GET /suppliers?type=&status=`, `POST /suppliers/{id}/contracts`, `GET /suppliers/{id}/contracts`,
  `POST /expenses` (→ `{id, financeEntryId, requiredDocuments[]}`), `POST /contracts/flag-expiring`. As
  escritas exigem `ROLE_FINANCE`. Erros novos: `admin.supplier.not-found` (404),
  `admin.supplier.invalid` (400), `admin.contract.invalid` (400), `admin.expense.invalid` (400),
  `admin.expense.duplicate` (409); i18n pt-BR + fallback en. OpenAPI **0.20.0**.

## Decisões (decision-log)

- **DL-0084** Admin é módulo Modulith próprio (22º), registro enxuto; **procurement = comprar**
  (fronteira) — Alta/Moderada.
- **DL-0085** mapa `kind`→`EntryType`→documento; `EntryType` +SERVICE/+OTHER_EXPENSE (aditivo); seed
  Compliance aditivo (sem editar V8) — Média/Barata.
- **DL-0086** integração por **fachada** `FinanceService.register` + **porta de leitura**
  `DocumentRequirementDirectory`; idempotência por UNIQUE `(supplier, period, kind)`; folha, acíclico —
  Alta/Moderada.
- **DL-0087** `AdminContractExpiring` por job de relógio controlado (30d, idempotente, alerta) —
  Média/Barata.
- **DL-0088** escritas exigem ROLE_FINANCE (403 + auditoria); alteração auditada via `system_audit`
  (`ADMIN_CHANGE`) — Média/Barata.

**Nenhuma** decisão desta fase é Confiança=Baixa **e** Reversibilidade=Cara: a Open Question de
procurement foi fechada pela própria spec (fronteira — comprar se exigido), e o resto segue padrões já
validados do projeto (genérico enxuto como Assets/Finance; relógio controlado; gate HTTP + auditoria).

## Riscos / o que fica para depois

- **Procurement completo** (cotação/aprovação/ordem de compra) é **comprar se exigido** (DL-0084): o
  cadastro de fornecedor/contrato/despesa é a base; um sistema de compras entra por cima (ou um pacote
  externo, com Admin virando adaptador). O seam (gerar lançamento + referenciar documento) sobrevive.
- **Confirmação/liquidação** da despesa (PROVISIONAL→CONFIRMED→SETTLED, PAYMENT_PROOF) são do
  Finance/Payout — Admin só gera o lançamento (BR4). O requisito AT_SETTLEMENT já está seedado.
- **Mapa final `kind`→`EntryType`→documento** (DL-0085, Confiança=Média) é confirmável com a
  contabilidade do cliente — a extensão é só nova migração de seed.
- **Tela Angular do Admin** fica para a Fase 10 (UX profissional); a entrega da SPEC-0025 é registro/API.
- **Fim do bloco 8x.** As próximas fases (9–15) são estruturais (achatar `internal`), UX/Frontend,
  observabilidade, E2E/Playwright, IdP externo (Fase 13) e upgrade de stack (Fase 14).
