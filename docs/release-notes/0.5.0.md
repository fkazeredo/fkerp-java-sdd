# Release 0.5.0 — Cancelamento como objeto + Armadilha do Merchant + No-Show

- **Data:** 2026-06-29
- **Fase do ROADMAP:** 4 (Cancelamento como objeto + armadilha do merchant)
- **Specs entregues:** SPEC-0010 (gradua o cancelamento simples da SPEC-0006)
- **Tag git:** `0.5.0`

## Destaques

O cancelamento deixa de ser "desfaz tudo, sem multa" e vira **política como objeto**: uma
`CancellationPolicy` (STANDARD | ALL_SALES_FINAL | CUSTOM, janelas de multa, `costBearer`,
`merchantOfRecord`) **congelada na Booking na confirmação** (snapshot, como a proveniência do Quote).
O cancelamento usa a política congelada, calcula a **multa** pela janela aplicável e materializa os
**encargos como fatos distintos**. A keystone é a **armadilha do merchant**: sob `ALL_SALES_FINAL` em
venda *merchant of record*, a Acme **continua devendo** ao fornecedor **mesmo reembolsando o cliente**
— **duas obrigações que NÃO se anulam**, modeladas e gravadas separadamente e **provadas por teste de
regressão**. Inclui a `NoShowPolicy` (fee + dispensa por prova de voo cancelado). Tudo no módulo
`booking` (sem módulo especulativo — DL-0020). **157 testes verdes**; sem UI nesta fase (backend-first,
como Fases 2–3).

## Adicionado / Alterado

- **Added — Política como objeto (fatia 9a):** `CancellationPolicy` (enum `CancellationType` com
  comportamento, `PenaltyWindow`, `CostBearer`, `NoShowPolicy`, `Charge`/`ChargeKind`); cálculo da
  multa pela janela aplicável (BR2/BR4, Money scale 2 HALF_UP); fonte administrável por `scope_ref`
  (`CancellationPolicySource`, `CancellationPolicyAdminService`) com `GET/PUT
  /api/products/{ref}/cancellation-policy`. `cancellation.policy.invalid` → 400.
- **Added — Cancelamento rico + armadilha do merchant (fatia 9b):** `BookingCancellationSnapshot`
  congelado na confirmação (BR1) com os valores de referência do Quote; `CancellationCharges`
  (calculador puro) que **acumula** encargos e **nunca** os compensa (DL-0024); `POST
  /api/bookings/{id}/cancel` agora `{reason, serviceStartsAt, refundAmount}` → `CancellationResult`;
  persistência em `cancellation_charges` (BR7); eventos `CancellationCharged` +
  `MerchantObligationIncurred`. ALL_SALES_FINAL merchant + reembolso → **SUPPLIER + CUSTOMER_REFUND**
  (costBearer ACME), que **não se anulam**.
- **Added — No-show (fatia 9c):** `POST /api/bookings/{id}/no-show` aceita `{flightCancelledProof}`;
  cobra a fee, **dispensa** com prova quando a política permite (BR6); evento `NoShowCharged`.
- **Changed:** `BookingService` agora congela o snapshot na confirmação e tem caminhos próprios para
  `cancel`/`noShow` (encargos); `Booking` ganhou `scopeRef`; `CreateBookingRequest`/`CancelBookingRequest`
  enriquecidos (retrocompatíveis com defaults nulos); `HttpErrorMapping` + `OpenApiConfig` + `pom.xml`
  para `0.5.0`.

## Migrações

- `V12__create_cancellation_policies.sql`: fonte da política por `scope_ref` (índice único).
- `V13__create_cancellation_snapshot_and_charges.sql`: `bookings ADD COLUMN scope_ref`;
  `booking_cancellation_snapshots` (política + no-show + valores de referência congelados);
  `cancellation_charges` (uma linha por encargo — BR5/BR11).

## Contratos / API

- `GET/PUT /api/products/{ref}/cancellation-policy` (papel administrativo) → 200 | 400
  `cancellation.policy.invalid`.
- `POST /api/bookings/{id}/cancel` — `{reason, serviceStartsAt, refundAmount?}` → 200
  `CancellationResult {bookingId, status, policyType, charges[]}` | 404 | 409.
- `POST /api/bookings/{id}/no-show` — `{flightCancelledProof?}` → 200 `NoShowResult {bookingId,
  status, charge?, waived}` | 404 | 409.
- OpenAPI via springdoc; erro estável `{code,message,fields}`; i18n pt-BR + fallback. O cancel/no-show
  **mudaram o corpo de resposta** (de `BookingView` para os envelopes de encargo) — é a graduação
  prevista da SPEC-0006; como há novos recursos e mudança de contrato compatível com o roadmap,
  permanece **MINOR** (ADR 0015).

## Decisões (decision-log)

- **DL-0020** — Cancelamento rico vive no módulo `booking` (sem módulo novo) — Alta/Moderada.
- **DL-0021** — Merchant of record é atributo por marca/contrato; default afiliado (costBearer=SUPPLIER);
  merchant=true → ACME — Alta/Moderada (adota a Recomendação do ROADMAP para Q3).
- **DL-0022** — Multa/encargos na moeda original, sem conversão cambial nesta fase — Média/Barata.
- **DL-0023** — No-show: dispensa por voo cancelado via flag de prova rastreável (conformidade =
  Compliance) — Média/Barata.
- **DL-0024** — Encargos são fatos distintos que **nunca** se compensam (a armadilha) — Alta/**Cara**.

## Riscos e pendências

- **Execução do reembolso/lançamento contábil:** `CancellationCharged`/`MerchantObligationIncurred`/
  `NoShowCharged` ficam in-process **sem consumidor** (Finance/Payout/Intelligence são fases futuras);
  viram contrato/outbox ao cruzar serviço.
- **Conversão cambial** da multa (DL-0022): adiada para a borda contábil; reversão barata e localizada.
- **Conformidade do documento de prova** do no-show (DL-0023): do Compliance; hoje a prova é um flag.
- **Não-compensação** dos encargos (DL-0024) é **Reversibilidade Cara**: sair do "sem netting" exige
  decisão de negócio e nova spec — é a tese econômica da fase.
- **Telas Angular:** backend-first nesta fase (acompanha a dívida das Fases 2–3).
