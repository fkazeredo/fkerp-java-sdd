# Release 0.16.0 — Assets: patrimônio interno (equipamentos, licenças) + alerta de vencimento

- **Data:** 2026-06-29
- **Fase do ROADMAP:** 8h (Assets — SPEC-0021, subdomínio **Supporting/Generic**)
- **Specs entregues:** SPEC-0021 (patrimônio interno: registro de equipamentos/licenças/outros bens +
  ciclo de vida + vínculos por valor a Compliance/Finance + alerta de licença a vencer)
- **Tag git:** `0.16.0`

## Destaques

A Fase 8h adiciona o **18º módulo Modulith**, `com.fksoft.domain.assets`, o **contexto de patrimônio
interno**: registra os **bens da própria Acme** — equipamentos, licenças de software e outros bens —
com um **ciclo de vida básico** (ACTIVE → RETIRED) e os **vínculos por valor** que o resto do sistema
precisa: o **documento de aquisição/contrato no Compliance** e o **lançamento de custo no Finance**
(ids como **valor**, nunca FK). É uma entrega **enxuta e real**: um registro que amarra custo↔documento
e alerta vencimento — **não** um sistema de gestão de ativos completo (sem depreciação contábil, sem
manutenção/chamados de TI, sem estoque de revenda — se o cliente exigir gestão plena, **comprar**,
DL-0065). Assets é **patrimônio, não produto**: não precifica venda nem entra no fluxo comercial (BR5).
Duas fatias:

1. **Registro + ciclo de vida (8h-1, V26, DL-0064/0065/0067/0068).** `Asset` tem `type`
   (EQUIPMENT | SOFTWARE_LICENSE | OTHER), `identifier`, `status` (nasce **ACTIVE**), `acquisitionDate`,
   `acquisitionCost` (Money) e, para **SOFTWARE_LICENSE**, um `expiresAt` **obrigatório** (BR1 — sem ele,
   400 `assets.license.expiry-required`). A aquisição referencia o **documento (Compliance)** e o
   **lançamento de custo (Finance)** por **valor** (BR2). A **baixa** (RETIRED) é **auditada**
   (quem/quando/motivo) e **terminal**: baixar de novo é 409 `assets.asset.already-retired`, preservando
   a 1ª auditoria (BR4/DL-0068). Registrar publica **`AssetRegistered`** in-process; Assets é
   **módulo-folha** — publica os eventos mas **não fia consumidores** em Finance/Intelligence (lançar
   custo automático de patrimônio é regra de negócio que a spec não define — DL-0067).
2. **Alerta de licença a vencer (8h-2, reusa V26, DL-0066).** Um job de **relógio controlado**
   (`AssetLicenseExpiryScheduler` → `flagExpiringLicenses(now)`), antecedência **30 dias**
   (configurável), varre as licenças ACTIVE com `expiresAt` no horizonte e publica
   **`AssetLicenseExpiring`** **uma vez** por licença (idempotente por `expiry_signaled_at`). É **alerta,
   não bloqueio**. A listagem `GET /api/assets?expiringWithinDays=N` devolve, ad-hoc, as licenças a
   vencer nos próximos `N` dias (default 30) — só licenças ACTIVE no horizonte.

## Regras de negócio (SPEC-0021)

- **BR1** `Asset` com type/identifier/status/acquisitionDate/acquisitionCost; **SOFTWARE_LICENSE** exige
  `expiresAt`. **BR2** aquisição referencia documento (Compliance) e lançamento (Finance) por valor, sem
  FK. **BR3** licença com `expiresAt` próximo publica `AssetLicenseExpiring` (alerta — não bloqueia).
  **BR4** baixa (RETIRED) auditada (quem/quando/motivo). **BR5** Assets **não** precifica venda nem entra
  no fluxo comercial (é patrimônio, não produto). Decisões assumidas BR6–BR10 (DL-0064..0068).

## Eventos

- `AssetRegistered {assetId, type, occurredAt}` — produtor `assets`. Consumidores previstos: `finance`
  (custo), `intelligence` (custo fixo/infra) — **seam** publicado, não fiado nesta fatia (DL-0067).
- `AssetLicenseExpiring {assetId, expiresAt, occurredAt}` — alerta de governança/TI.

## Migração

- **V26__create_assets.sql** (idempotente): tabela `assets` (custo como `acquisition_cost`+`currency`;
  `document_id`/`finance_entry_id` **valores**; `expiry_signaled_at` para a idempotência do alerta;
  `retired_at`/`retired_by`/`retirement_reason` da baixa). Índices: `ix_assets_type_status` (listagem) e
  `ix_assets_license_expiry` **parcial** (candidatos do sweep). Nenhuma FK cross-contexto.

## Contratos / OpenAPI

- Novos endpoints sob `/api/assets`: `POST /api/assets`, `GET /assets/{id}`,
  `GET /assets?type=&status=&expiringWithinDays=`, `POST /assets/{id}/retire`,
  `POST /assets/flag-expiring`. OpenAPI **0.16.0**. Erros `assets.asset.not-found` (404) /
  `assets.asset.invalid` (400) / `assets.license.expiry-required` (400) /
  `assets.asset.already-retired` (409), i18n pt-BR + fallback en.

## Decisões (decision-log)

- **DL-0064** Assets é contexto **separado** de Portfolio (Q2: dois contextos); registro enxuto; 18º
  módulo (Alta/Moderada).
- **DL-0065** **sem depreciação/gestão plena** (Out of Scope): registro + seam comprar-vs-construir
  (Alta/Moderada).
- **DL-0066** alerta de licença a vencer por **job de relógio controlado** (30d, idempotente)
  (Média/Barata).
- **DL-0067** Assets é **folha**: publica `AssetRegistered`/`AssetLicenseExpiring`; **não** fia
  consumidores Finance/Intelligence (Média/Barata).
- **DL-0068** baixa **auditada inline**; ACTIVE→RETIRED terminal; re-baixar → 409 (Média/Barata).

Nenhuma decisão desta fase é Confiança=Baixa **e** Reversibilidade=Cara: a Q2 foi resolvida pela
recomendação do arquiteto e o resto segue padrões já validados do projeto.

## Riscos / o que fica para depois

- **Custo automático no Finance e insight de custo fixo no DSS** ficam como **seam** (eventos
  publicados, sem consumidor — DL-0067): quando o dono definir a política de custo de patrimônio,
  adiciona-se um listener no módulo consumidor sem tocar em Assets.
- **Depreciação / manutenção / estoque de revenda** = **comprar** um sistema dedicado (DL-0065); este
  módulo permanece o registro/seam.
- **Tela Angular** do Assets não entra nesta fase (módulo de back-office, como 8c–8g) — fica para a
  Fase 10 (UX).
