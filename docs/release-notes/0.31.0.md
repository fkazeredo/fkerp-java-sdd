# Release 0.31.0 — Fase 18c · enums de referência → cadastros (Sourcing/Exchange/Booking/Compliance) + rótulos nas telas

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 18 (fatia 18c de 4)
- **Specs entregues:** **SPEC-0031** (atualizada — tipos 18c marcados como entregues)
- **Decisões:** **ADR-0019** (padrão enum→cadastro) + **DL-0117** (enums Sourcing/Exchange/Booking/Compliance)
- **Tag git:** `0.31.0`

## Destaques

Terceira fatia da Fase 18: reusa o módulo `cadastro` (18a) e o pipe de rótulo (18b) e converte mais
**quatro grupos** de enums de referência para **`code` validado**, mantendo o contrato de fio
inalterado (**`code` = nome do antigo constante do enum ⇒ JSON idêntico**). As telas
Sourcing/Exchange-desk/Booking(Cancelamento)/Compliance passam a mostrar o **rótulo** do cadastro em
vez do código.

- **Sourcing:** `OfferOrigin`, `IntegrationLevel` → `code` validado (`OFFER_ORIGIN`,
  `INTEGRATION_LEVEL`), validados na escrita (`register`). O ramo **INTEGRATED** do quoting (DL-0018)
  continua cunhando `INBOUND` via `IntegrationLevelCodes`.
- **Exchange:** `MarketRateSource` → `code` (`MARKET_RATE_SOURCE`). É **produzido pelo sistema** (o
  controller de contingência grava `MANUAL`), então vira cadastro para rótulo — sem validação de
  escrita.
- **Booking:** `ChargeKind`, `CancellationType` → `code` (`CHARGE_KIND`, `CANCELLATION_TYPE`),
  `CancellationType` validado no PUT da política. A **armadilha do lojista** (ALL_SALES_FINAL: custo do
  fornecedor + reembolso do cliente que **nunca se compensam**) e as **janelas de multa** são
  preservadas por `CancellationTypeCodes`/`ChargeKindCodes` (DL-0024/DL-0010).
- **Compliance:** `DocumentType`, `SignedFormat`, `RequirementPhase` → `code` (`DOCUMENT_TYPE`,
  `SIGNED_FORMAT`, `REQUIREMENT_PHASE`), `DocumentType` validado no `upload`. A **retenção legal**
  (FISCAL 5 anos / CONTRACT 10 anos) e o **close-check** por `AT_REGISTRATION` (DL-0012) são
  preservados. `SignedFormat` é produzido pelo adaptador ingestor — sem validação de escrita.

## Adicionado

- **Constantes `*Codes`** (comportamento cablado, DL-0117): `OfferOriginCodes`/`IntegrationLevelCodes`
  (procedência do ACL + ramo INTEGRATED), `MarketRateSourceCodes` (FEED/MANUAL),
  `ChargeKindCodes`/`CancellationTypeCodes` (postagem AP/AR + `usesWindows`/`isAllSalesFinal`),
  `DocumentTypeCodes`/`SignedFormatCodes`/`RequirementPhaseCodes` (retenção + close-check).
- **Rótulo nas telas** via `CadastroLabelPipe` (18b): Sourcing (origem/nível), Exchange-desk (source),
  Booking/Cancelamento (tipo), Compliance (documento/formato assinado). +8 tipos no
  `cadastro.models` do frontend.
- **Migração V35** semeia os valores atuais dos 8 tipos (37 itens; `code`=nome do enum, `label`
  pt-BR), idempotente (`ON CONFLICT DO NOTHING`).

## Alterado

- **Sourcing/Exchange/Booking/Compliance:** campos de enum passaram a `String code` (entidades, DTOs,
  views, eventos, repositórios, serviços; params de controller/`@RequestParam` `String`). O `switch`
  exaustivo do `BookingChargeEventsListener` (Finance) sobre `ChargeKind` virou `switch` sobre
  `ChargeKindCodes` com `default` seguro. O evento `LegalTimeRecordArchived` (People) deixou de
  importar `compliance.DocumentType` (passa a `String`) — remove uma dependência de tipo cross-módulo.
- **Telas Sourcing/Exchange-desk/Booking(Cancelamento)/Compliance** exibem o **rótulo** do cadastro
  (não o code). A severidade das tags continua usando o code cru.
- **OpenAPI** (versão → 0.31.0): os campos convertidos mantêm o schema `string` — **nenhum contrato
  `/api` existente mudou**.

## Removido

- Os enums `OfferOrigin`/`IntegrationLevel` (sourcing), `MarketRateSource` (exchange),
  `ChargeKind`/`CancellationType` (booking), `DocumentType`/`SignedFormat`/`RequirementPhase`
  (compliance) — substituídos por `code` validado + constantes `*Codes`.

## Migrações

- **V35__seed_cadastro_sourcing_exchange_booking_compliance.sql** — semeia os 8 tipos 18c (37 itens)
  em `cadastro_item`. Idempotente; nunca edita uma migração aplicada.

## Impacto em OpenAPI / contratos

- Os campos convertidos continuam `string` no JSON — **nenhum contrato existente quebrou** (invariante
  DL-0117). Sem novos endpoints. Versão do contrato → **0.31.0**.

## Riscos e o que fica para a fatia final (18d)

- **Código novo (não cablado):** um `OFFER_ORIGIN`/`DOCUMENT_TYPE`/`CHARGE_KIND` novo flui como dado
  puro — sem ramificação cablada (documento novo cai na retenção FISCAL padrão; charge-kind novo não
  posta no Finance) até uma fatia futura cablar — seam documentado (DL-0117).
- **`MarketRateSource`/`SignedFormat` não são validados na escrita** (produzidos pelo
  sistema/adaptador; `CAdES_P7S` é misto e colidiria com o validador que normaliza para maiúscula).
- **Sem trava de obrigatoriedade** (desativar o último item ativo de um tipo) — Open Question da
  SPEC-0031, adiada.
- **18d** converte os grupos restantes (People/Payout/AfterSales…) reusando o módulo, uma fatia por
  grupo — a fatia final da Fase 18.
