# Release 0.19.0 — Identity: autenticação real + papéis/permissões + auditoria de acesso (gradua o stub)

- **Data:** 2026-06-30
- **Fase do ROADMAP:** 8k (Identity — SPEC-0024, subdomínio **genérico**)
- **Specs entregues:** SPEC-0024 (autenticação real; modelo de papéis/permissões; enforcement por
  papel das ações sensíveis; auditoria de acesso — graduando o `DevStubUserContextProvider`)
- **Tag git:** `0.19.0`

## Destaques

A Fase 8k **gradua o stub de identidade** em **autenticação real**, com o **backend como única
autoridade** de autorização (security.md). É o **21º módulo Modulith** (`com.fksoft.domain.identity`,
DL-0080). Três fatias:

1. **Autenticação real + login JWT (8k-1, BR1/BR4, DL-0079/0080/0081).** Spring Security entra ligado;
   o ERP autentica **in-house** e é **Resource Server do seu próprio emissor JWT (HS256)**:
   `POST /api/identity/login` verifica a senha (**BCrypt**) de um usuário local e devolve um **token
   Bearer** com `userId`, `username` e `roles`. As requisições seguintes mandam `Authorization: Bearer
   <jwt>`; um filtro verifica o token e popula o contexto. O **`UserContextProvider` real**
   (`JwtUserContextProvider`) resolve usuário/roles do token — **graduando o stub sem mudar a porta**
   que os módulos consomem. Credencial inválida → **401 genérico** que **não revela** se o usuário
   existe (BR4). O **IdP externo OIDC vivo** (JWKS/rotação, escopos finos) é a **Fase 13** — a porta é
   o seam que sobrevive à troca.
2. **Papéis/permissões + auditoria de acesso (8k-2, BR2/BR3/BR5, DL-0082/0083).** Modelo
   `Role`/`Permission` (catálogo fechado) como **única fonte de verdade** de autorização interna. As
   **ações sensíveis** já citadas nas specs passam a **exigir o papel**: emitir NF → `ROLE_FINANCE`,
   fechar período → `ROLE_FINANCE`, disparar job / custodiar certificado → `ROLE_IT`, diretiva →
   `ROLE_DIRECTOR`, regra → `ROLE_DIRECTOR`/`ROLE_POLICY_ADMIN`. Negação → **403 auditado**; o
   enforcement é HTTP (Spring Security `hasRole`) **e** reafirma a checagem de domínio já existente
   (CommercialPolicy/DL-0038), que agora lê os papéis do **token real**. **Login e negação** são
   gravados na auditoria append-only do Platform (`system_audit`, `AUTH_LOGIN`/`ACCESS_DENIED`) —
   **nunca** a senha/token (BR4); reusa o seam do 8j em vez de uma tabela `access_audit` nova (Regra
   Zero, DL-0083). `GET /api/identity/roles` e `GET /api/identity/access-audit` (autorização TI/diretor).
3. **Login no frontend (8k-3).** Tela de **login** (Angular), `AuthService` (guarda o token, espelha o
   usuário do token), **interceptor** que anexa o Bearer e reage ao 401 (limpa sessão → login),
   **guard** de rota e o usuário/“Sair” no topo. O backend continua a autoridade — o front só espelha.

## Não quebrar o build (a maior restrição da fase)

Os **434 testes pré-existentes assumiam um ator autenticado com acesso total**. Graduar **≠**
rip-and-replace: a segurança fica **montada, não removida**. No profile `test`, uma `TestSecurityConfig`
roda a **mesma cadeia real** de autorização, mas autentica um **ator de teste com acesso total**
**somente** quando não há header `Authorization` — então os testes antigos seguem verdes **sem mandar
token**, enquanto os **testes de segurança** (que mandam token) exercitam o **401/403 reais**. O
`DevStubUserContextProvider` fica atrás do profile `dev`; em produção/default vale o `JwtUserContextProvider`
(BR6/DL-0081). Nenhum teste foi apagado/enfraquecido.

## Regras de negócio (SPEC-0024)

- **BR1** usuário/roles derivados de token verificado; `UserContextProvider` real substitui o stub sem
  mudar a interface. **BR2** autorização por papel; ação sensível exige o papel; negação → 403 + audit.
  **BR3** login/negação auditados. **BR4** sem segredo logado; erro de auth genérico. **BR5** modelo de
  papéis é a única fonte de autorização interna. **BR6** stub só em dev, off em produção.
  **BR7..BR11** ASSUMIDO (ver DL-0079..0083).

## Eventos

- `UserAuthenticated {userId, username, at}` — produtor `identity` (login bem-sucedido).
- `AccessDenied {actor, action, resource, at}` — produtor `identity` (negação 403).

## Migração

- **V29__create_identity.sql** (idempotente): `roles` (PK nome) + `role_permissions` (catálogo fechado),
  `identity_users` (BCrypt `password_hash` **só hash**, username **único**, status, version) +
  `user_roles` (M:N). **Seed** dos 6 papéis base + as permissões nomeadas das ações sensíveis. Usuários
  **dev/test** (um por papel, senha `dev12345`) são semeados **programaticamente** pelo `DevUserSeeder`
  **apenas** nos profiles dev/test — nunca em produção (BR6), sem hash hardcoded. A auditoria de acesso
  reusa `system_audit` (V28/Platform) — **sem tabela nova** (DL-0083).

## Contratos / OpenAPI

- Novos endpoints `/api/identity`: `POST /login` (público), `GET /me`, `GET /roles`,
  `GET /access-audit`. Esquema de segurança **Bearer JWT** documentado. Ações sensíveis passam a exigir
  o papel (403 sem ele). Erros novos: `identity.credentials.invalid` (401), `auth.unauthenticated`
  (401), `access.denied` (403), i18n pt-BR + fallback en. OpenAPI **0.19.0**.

## Decisões (decision-log)

- **DL-0079** auth real **in-house** (Spring Security + JWT HS256) no 8k; **OIDC externo vivo na Fase
  13**; porta `UserContextProvider` é o seam (**Baixa**/**Cara**) — **destacada**: comprar/qual IdP é
  Open Question do dono; trocar o emissor in-house por um IdP externo vivo é refator amplo.
- **DL-0080** novo módulo `domain.identity` (21º) + usuário local mínimo (V29); auditoria reusa
  `system_audit` (Alta/Moderada).
- **DL-0081** gradua o stub: porta intacta; JWT em prod; stub atrás de profile `dev`/`test`;
  `TestSecurityConfig` mantém os 434 testes verdes com a segurança montada (Alta/Barata).
- **DL-0082** modelo papel→permissão + mapa das ações sensíveis; enforcement HTTP + reafirma a checagem
  de domínio (Média/Moderada).
- **DL-0083** auditoria de acesso reusa o `system_audit` (Platform/8j); sem tabela nova (Alta/Moderada).

**DL-0079** é a única Confiança=Baixa **e** Reversibilidade=Cara da fase (a escolha do IdP só o dono
fecha); as demais são reversões baratas/moderadas e localizadas.

## Riscos / o que fica para depois

- **IdP externo OIDC vivo** (Keycloak/Entra/Cognito) é a **Fase 13** (DL-0079, Confiança=Baixa): o 8k
  entrega o modelo real + papéis + auditoria com emissor JWT in-house; a 13 consolida o Resource Server
  profissional (JWKS/rotação de chave, **refresh/silent-refresh** no front, **escopos finos**
  `crm:lead:read:all` → perfis) e move a gestão de usuários para o IdP. A porta `UserContextProvider` e
  o modelo de papéis sobrevivem à troca; o emissor/verificador/login não (reversão Cara).
- **Segredo de produção:** `IDENTITY_JWT_SECRET` (>= 32 bytes) **deve** ser definido em produção; o
  default de dev não é seguro. Usuários dev seed só existem nos profiles dev/test.
- **Mapeamento de toda ação sensível futura** consolida à medida que novas ações surgem (a spec assim
  pede; `role_permissions` é dado governável, DL-0082).
- **Tela de gestão de usuários/papéis** e o **silent-refresh** ficam para a Fase 10/13 (UX/AuthZ
  profissional). **Métricas** `auth_logins_total`/`access_denied_total` ficam para a Fase 11
  (observabilidade); aqui o fato vive na auditoria e nos logs de segurança.
