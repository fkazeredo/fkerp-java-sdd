# Release 0.23.0 — Fase 13 · Identity/AuthZ profissional (gradua a SPEC-0024)

- **Data:** 2026-06-30
- **Fase do ROADMAP:** 13
- **Specs entregues:** SPEC-0024 (**graduada** — não é spec nova)
- **Tag git:** `0.23.0`

## ⚠️ Mudança incompatível (breaking — destacada por ADR 0015 §4)

O endpoint de login próprio do 8k — **`POST /api/identity/login`** — foi **removido**. A autenticação
deixou de ser **in-house** (o ERP emitia/conferia o próprio JWT HS256) e passou a ser **delegada a um
IdP OIDC externo (Keycloak)**. Quem consumia `/api/identity/login` para obter um token deve agora
**obter o token no provedor** (Authorization Code + PKCE no frontend; o endpoint de token do IdP para
integrações). Em `0.y.z` o SemVer permite breaking, mas a quebra é **destacada** aqui.

## Destaques

A Fase 13 **consolida e gradua a SPEC-0024**: o ERP vira um **OAuth2 Resource Server** que valida os
**JWTs de um IdP OIDC externo (Keycloak) por JWKS** (RS256, **rotação de chave**), mapeando os
**papéis do realm** (`realm_access.roles`) para as autoridades Spring — o **modelo de papéis e a porta
`UserContextProvider` sobrevivem** à troca, só muda a *fonte* do token. O login no frontend passa a ser
**OIDC code+PKCE** com **silent-refresh real (refresh token)**. Isto **resolve as duas dívidas
diferidas**: **DL-0079** (IdP externo vivo) e **DL-0092** (silent-refresh real).

## Adicionado / Alterado / Removido

- **Added:** **Resource Server por JWKS** (`spring.security.oauth2.resourceserver.jwt.issuer-uri` +
  `jwk-set-uri`) validando o token do IdP externo (assinatura RS256, `iss`, `exp`), com **rotação de
  chave** automática (DL-0104).
- **Added:** mapeamento **`realm_access.roles` → autoridades `ROLE_*`** (preserva o catálogo de papéis
  da SPEC-0024) + exposição de `scope` como `SCOPE_*` para refino futuro (DL-0104).
- **Added:** **IdP de desenvolvimento Keycloak** no `docker-compose.yml` e no `compose.e2e.yaml`, com
  realm `acme` importado (`infra/keycloak/realm-acme.json`): 6 papéis, **aplicativo SPA público
  (PKCE + refresh)**, um **cliente direct-grant de E2E** (só teste) e **usuários seed** (um por papel +
  `dev`, senha `dev12345` — **dev/E2E only**) (DL-0103).
- **Added:** **frontend OIDC** com `angular-oauth2-oidc` — botão **"Entrar com SSO"** (code+PKCE) e
  **silent-refresh real** por refresh token (gradua DL-0092/DL-0106); interceptors de correlation-id e
  de auth passam a **ignorar chamadas cross-origin** (CORS do IdP).
- **Added:** caminho de **teste com JWKS local** (`TestJwtTokens`, par RSA de teste) — os testes mintam
  tokens RS256 no formato do Keycloak e exercitam o decoder JWKS real **sem IdP na internet** (DL-0105);
  o `TestSecurityConfig` mantém o ator full-access quando não há `Authorization` (suíte verde).
- **Changed:** `GET /api/identity/me` registra o **`AUTH_LOGIN`** (primeiro toque autenticado) no
  `system_audit` (a auditoria de login migra do `/login` removido); `GET /roles` e `GET /access-audit`
  inalterados.
- **Changed:** esquema de segurança da **OpenAPI** → **bearer JWT (OIDC, JWKS)**; versão 0.22.1 →
  **0.23.0** (pom.xml + OpenAPI) e descrição com a frase da Fase 13.
- **Removed (breaking):** `POST /api/identity/login`, o emissor in-house (`JwtIssuer`/HS256), o store
  **local de usuários** (`identity_users`/`user_roles`, hash BCrypt), o `DevUserSeeder`, o
  `DevStubUserContextProvider` e o `PasswordHasher`/`BCryptPasswordHasher` (DL-0105/DL-0107). O ERP
  **deixa de custodiar senha**.

## Migrações

- **V31__retire_local_user_store.sql** — `DROP TABLE IF EXISTS user_roles, identity_users` (idempotente).
  **Mantém** `roles`/`role_permissions` (o catálogo papel→permissão continua **local**, fonte do
  enforcement — BR16/DL-0107). Os usuários vivem no IdP.

## Contratos / API

- **Removido (breaking):** `POST /api/identity/login`.
- **Mantidos:** `GET /api/identity/me`, `GET /api/identity/roles`, `GET /api/identity/access-audit`.
- **Esquema de segurança:** Bearer JWT do **IdP externo (OIDC/JWKS)**, documentado na OpenAPI.
- Ações sensíveis continuam exigindo o papel (Financeiro/TI/Diretor/Curador) → 403 + auditoria; token
  ausente/inválido → 401 genérico.

## Decisões (decision-log)

- **DL-0103** — IdP de dev = **Keycloak** (realm importado + client PKCE + papéis + usuários seed).
  **Baixa / Cara** (qual IdP em produção é decisão do dono).
- **DL-0104** — Resource Server por **JWKS** + `realm_access.roles`→autoridades. **Média / Moderada.**
- **DL-0105** — Caminho de teste com **JWKS local** + **remoção** de `POST /login` (breaking).
  **Média / Moderada.**
- **DL-0106** — Frontend **OIDC code+PKCE** + **silent-refresh real**. **Média / Moderada.**
- **DL-0107** — Catálogo papel→permissão **fica local**; store de **usuários aposentado** (V31).
  **Alta / Moderada.**
- **Fechadas:** **DL-0079** (IdP externo vivo) e **DL-0092** (silent-refresh real) → **RESOLVIDAS**.

## Como rodar / verificar

```bash
cd backend && ./mvnw verify                 # 476 testes, gates verdes (ArchUnit/Modulith/Spotless/Checkstyle/JaCoCo)
cp .env.example .env
docker compose up --build                   # app + db + KEYCLOAK (8088) + observabilidade
# Keycloak: http://localhost:8088 (admin/admin) — realm 'acme' importado
# App: http://localhost:4200 → "Entrar com SSO" → login no Keycloak (ex.: director/dev12345)

# E2E (stack isolado, Keycloak na 8089):
cd frontend && npm run e2e:up && E2E_BASE_URL=http://localhost:4201 npm run e2e && npm run e2e:down
# 11 specs Playwright verdes: login OIDC real → painel; 401/403 por papel contra tokens do IdP.
```

## Riscos e pendências

- **IdP de produção é decisão do dono** (DL-0103, Confiança=Baixa): o Keycloak entregue serve dev/E2E;
  trocar por Entra/Cognito/Keycloak self-hosted é **configuração** (issuer/clientId), não código — a
  porta `UserContextProvider` e os papéis sobrevivem.
- **Issuer × JWKS em contêiner:** o token carrega o issuer **público** (o que o navegador usa), mas o
  backend busca as chaves pela **rede interna** (`OIDC_JWK_SET_URI` = serviço `keycloak`) — evita a
  armadilha de `localhost` dentro do contêiner. Em produção, alinhe ambos ao host real do IdP.
- **Usuários/seed são dev/E2E only** (senha fraca conhecida): nunca em produção.
- **Autorização fina por escopo** (`crm:lead:read:all`-style) fica **disponível** (`SCOPE_*` expostos)
  mas o enforcement atual segue **por papel** (Regra Zero) — refina-se quando uma ação nova exigir.
