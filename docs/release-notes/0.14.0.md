# Release 0.14.0 — Marketing: consentimento LGPD, campanha segmentada e atribuição

- **Data:** 2026-06-29
- **Fase do ROADMAP:** 8f (Marketing — SPEC-0019, subdomínio **Supporting**)
- **Specs entregues:** SPEC-0019 (campanha/segmentação/newsletter + consentimento LGPD + atribuição)
- **Tag git:** `0.14.0`

## Destaques

A Fase 8f adiciona o **16º módulo Modulith**, `com.fksoft.domain.marketing`, o **contexto de
marketing B2B** que trata o **consentimento LGPD como cidadão de primeira classe**: nenhuma campanha
sai sem consentimento, a base é segmentada sobre dados que já existem (minimização), o envio externo é
uma **ACL** (porta `NewsletterSender` com mock rastreável) e a **atribuição** liga a campanha à reserva
para alimentar o DSS. **Não é um CRM** — é a camada de consentimento/atribuição (CRM pleno = comprar).
Três fatias:

1. **Consentimento (log append-only) (8f-1, V24, DL-0055/DL-0056).** O agregado `Consent` é uma
   **linha imutável**: conceder e revogar **apendam** uma linha; o **estado atual** é a última linha por
   `(titular, finalidade)`; o histórico é preservado; reconsentir após revogar volta a `GRANTED`. O
   modelo é **single opt-in** no v1 (comporta double opt-in sem refator). Endpoints
   `POST /api/marketing/consents`, `DELETE /consents/{id}` (revoga), `GET /consents` (estado + histórico).
   Erros nunca vazam o id do titular (LGPD).
2. **Segmentação + campanha + disparo (8f-2, DL-0059/DL-0055).** `Segment` guarda o critério em
   `criteria_json` (jsonb) **validado contra um catálogo fechado** de campos (minimização — campo
   desconhecido → 400). `Campaign` tem um `code` público único (atribuição). O **disparo filtra por
   consentimento ANTES de enfileirar** (BR2): só titulares com `GRANTED` vigente entram; os demais são
   **suprimidos e contados** (não é erro global); é **idempotente por `(campaignId, recipient)`** (BR4)
   — reenvio não duplica. O envio externo passa pela porta `NewsletterSender` (mock rastreável em
   `infra.integration.newsletter`); o **DTO do provedor não cruza para o domínio** (regra ArchUnit nova,
   ADR 0007). Endpoints `POST /segments`, `GET /segments/{id}/preview`, `POST /campaigns`,
   `POST /campaigns/{id}/send`.
3. **Atribuição + conversão + exclusão LGPD (8f-3, DL-0057/DL-0058).** `Attribution` liga um
   **código→reserva** (UNIQUE) por um intake próprio do Marketing; ao receber **`BookingConfirmed`** com
   código pré-registrado, **confirma a conversão** e publica **`CampaignConverted`** para a Intelligence
   (consumidor-folha — a regra "aconselha, nunca comanda" segue verde). O evento `BookingConfirmed`
   **não** foi alterado (seam para UTM nativo no Booking, quando existir). A **exclusão LGPD**
   (`POST /erasure`) remove a PII de marketing e **anonimiza** o log, **preservando um tombstone de
   revogação** (pseudônimo SHA-256) para a supressão futura; as `attributions` (sem PII) e outras bases
   legais (fiscal/financeiro/reserva) **permanecem intactas**. Endpoints `POST /attribution`,
   `GET /attribution`, `POST /erasure`.

## Regras de negócio (SPEC-0019)

- **BR1** Consentimento registra titular/finalidade/base legal/status/origem/timestamp; revogação é
  append (histórico preservado). **BR2** Nenhum envio sem `GRANTED` para a finalidade (suprime+conta).
  **BR3** Segmento só sobre dados existentes (minimização). **BR4** Idempotência de envio por
  `(campanha, destinatário)`. **BR5** Atribuição liga código→reserva e publica `CampaignConverted`.
  **BR6** Exclusão LGPD atendível preservando o que outra base legal obriga a manter.

## Eventos

- `CampaignSent {campaignId, targeted, suppressed, occurredAt}` — produtor `marketing`.
- `CampaignConverted {campaignId, bookingId, occurredAt}` — consumidor `intelligence` (atribuição).
- `ConsentRevoked {subjectRef, subjectType, purpose, occurredAt}` — supressão futura.

## Migração

- **V24__create_marketing.sql** (idempotente): `consents` (log append-only + índice por
  `(subject, purpose, created_at DESC)`), `segments` (`criteria_json` jsonb), `campaigns`
  (`code` único), `campaign_sends` (PK `(campaign_id, recipient_ref)` — idempotência) e `attributions`
  (UNIQUE `(campaign_code, booking_id)`). Nenhuma FK cross-contexto (ids de outros contextos são valor).

## Contratos / OpenAPI

- Novos endpoints sob `/api/marketing/*` (consents, segments, campaigns, attribution, erasure).
  OpenAPI atualizada (versão **0.14.0**). Erros `marketing.consent.*` / `marketing.segment.invalid`
  (400) / `marketing.campaign.not-found` (404) / `marketing.newsletter.failure` (502), i18n pt-BR +
  fallback en, sem PII.

## Decisões (decision-log)

- **DL-0055** porta `NewsletterSender` (ACL) + mock rastreável; **single opt-in** no v1 (Média/Moderada).
- **DL-0056** `Consent` append-only; estado = última linha por `(titular, finalidade)` (Alta/Moderada).
- **DL-0057** atribuição por **intake próprio** + confirmação na `BookingConfirmed`; **não** altera o
  evento do Booking (Média/Moderada).
- **DL-0058** exclusão LGPD remove PII mas **preserva tombstone de revogação anonimizado** — _Confiança
  **Baixa** / Reversibilidade **Cara**_ (alcance exato = decisão de DPO; o expurgo é destrutivo).
- **DL-0059** `Segment` `criteria_json` validado por catálogo fechado; fronteira **"não é CRM"**
  (Média/Moderada).

## Riscos / o que fica para depois

- **DL-0058 (Baixa/Cara):** o alcance exato do apagamento LGPD precisa de confirmação do DPO antes do
  primeiro uso real (expurgo é destrutivo).
- **UTM nativo no Booking** (DL-0057) e **double opt-in / provedor real** (DL-0055) são evoluções
  aditivas adiadas. O **read-model de ROI de campanha** (PromoConversion, 8.2-F) é território da
  Intelligence (Fase 7) — aqui entregamos só o **sinal** de conversão.
