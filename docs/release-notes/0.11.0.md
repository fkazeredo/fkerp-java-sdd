# Release 0.11.0 — Billing: NFS-e de comissão, ISS por regime e arquivamento fiscal

- **Data:** 2026-06-29
- **Fase do ROADMAP:** 8c (Billing — SPEC-0016, subdomínio **Supporting**)
- **Specs entregues:** SPEC-0016 (NFS-e sobre a comissão + ISS/retenções)
- **Tag git:** `0.11.0`

## Destaques

A Fase 8c adiciona o **13º módulo Modulith**, `com.fksoft.domain.billing` (módulo **folha**), que
emite a **Nota Fiscal de Serviço (NFS-e) sobre a comissão** — a receita real da Acme — e materializa a
tese econômica do redesenho (3.2/7.7): **o imposto incide só sobre a comissão**, nunca sobre o valor do
pacote que "apenas passa". Três capacidades:

1. **Cálculo de tributos parametrizado por regime, atrás de estratégia trocável (Q7 / DL-0044).** O
   ISS = `alíquota(município) × base_comissão` (escala 2 HALF_UP, kernel `Money`); o **regime default
   é Simples Nacional** (sem retenções federais — IN RFB 1.234/2012) e o **emitente é a própria Acme**.
   A `TaxRegimeStrategy` é uma porta: trocar o regime real do contador (Presumido/Real) = plugar outra
   estratégia, **sem refatorar** o agregado nem o fluxo (provado por um swap-test). A **base = comissão
   (BR1)** é regressão: o agregado **não tem campo** para o pacote.
2. **Emissão via ACL com mock rastreável (DL-0046) + arquivamento + lançamento (DL-0047).** O
   webservice municipal de NFS-e é uma **porta de domínio** (`NfseGateway`) com adaptador
   `infra.integration.nfse` — o **DTO do vendor não vaza** para o domínio (regra ArchUnit, como nas
   Fases 3 e 6). A assinatura e-CNPJ é a porta `CertificateSigner` (stub → Platform/SPEC-0023). Uma
   emissão bem-sucedida é **arquivada no Compliance** (documento anexado ao lançamento de comissão,
   satisfazendo o `DocumentRequirement` — o mês passa a poder fechar) e o **ISS é lançado no Finance**
   (`TAX_PAYABLE`) consumindo o evento `CommissionInvoiceIssued` **idempotente** (`finance → billing`
   acíclico, mesmo padrão da 8b). Falha do webservice é **classificada** (REJECTED → 422;
   TIMEOUT/UNAVAILABLE → 502) — **nunca uma "emitida" falsa** (BR7). **Idempotência por comissão** (BR4)
   via índice UNIQUE parcial.
3. **API REST.** `POST /api/billing/invoices` (rascunho), `/{id}/issue`, `/{id}/cancel` (BR6),
   `GET /{id}`, com erros estáveis e OpenAPI atualizada.

**265 testes verdes**, **13 módulos Modulith acíclicos** (novas arestas `finance → billing` por evento
e `infra.nfse → billing/compliance`, sem ciclo: `billing` é folha), 0 violações de Checkstyle; sem UI
nesta fase (a tela de Billing é da iniciativa UX, SPEC-0026).

  `document_id` como **valor**, sem FK cross-contexto) com **UNIQUE parcial** `(commission_entry_id)

## Contratos / API

- `POST /api/billing/invoices` → 201 `{id, base, status:RASCUNHO, ...}`.
- `POST /api/billing/invoices/{id}/issue` → 200 `{status:EMITIDA, number, verificationCode, iss,
  documentId}` | 422 `billing.municipality.rejected` | 502 `billing.nfse.webservice-failure`.
- `POST /api/billing/invoices/{id}/cancel` → 200 `{status:CANCELADA}`.
- `GET /api/billing/invoices/{id}` → 200 | 404 `billing.invoice.not-found`. Reemitir sem cancelar →
  no-op idempotente (mesma nota). **Endpoints novos / módulo novo** → **MINOR** (ADR 0015): após
  0.10.0, **0.11.0**. OpenAPI via springdoc (code-first); erro estável `{code, message, fields}`.

## Decisões (decision-log)

- **DL-0044** — Q7: regime **Simples Nacional** default + emitente = Acme; ISS/retenções parametrizados
  por regime+município atrás de `TaxRegimeStrategy` trocável — **Confiança=Baixa, Reversibilidade=Cara**
  (incógnita de negócio: só o contador fecha o regime real; o impacto fiscal de notas já emitidas é
  externo e caro). **Levar Q7 ao contador antes de emitir em produção.**
- **DL-0045** — Novo módulo folha `domain.billing`; `CommissionInvoice`; **base tributável = comissão**
  (sem campo de pacote); referência ao lançamento por id+valor, sem FK — **Alta / Moderada**.
- **DL-0046** — NFS-e como porta `NfseGateway` + adaptador ACL com **mock rastreável**; e-CNPJ via
  `CertificateSigner` (stub → SPEC-0023); falha classificada 422/502 — **Média / Moderada**.
- **DL-0047** — Emissão idempotente (UNIQUE parcial); arquiva no Compliance (orquestrador infra, devolve
  `documentId` síncrono); Finance lança o tributo por evento; `billing` folha → grafo acíclico —
  **Média / Moderada**.

> **decision-log/INDEX.md:** **DL-0044 é a sensível desta fase** — Confiança=Baixa **e**
> Reversibilidade=Cara (Q7, regime tributário). DL-0045/0046/0047 não são barata-só, mas sem unknown de
> negócio.

## Riscos e pendências

- **Q7 (regime tributário) é incógnita de negócio (DL-0044):** o default Simples Nacional vale até o
  contador confirmar. Trocar para Presumido/Real é plugar a estratégia (sem refator de domínio), mas
  notas já emitidas sob o regime anterior só se corrigem por **cancelamento + reemissão** (imutáveis).
  **Confirmar com o contador antes de produção.**
- **Município(s) e padrão de NFS-e (ABRASF/Nacional):** parametrizados no adaptador (mock rastreável);
  o webservice municipal real e o XML completo ficam fora de escopo (DL-0046) — trocar o adaptador, não
  o domínio.
- **Custódia real do e-CNPJ:** porta `CertificateSigner` com stub; dono = **Platform (SPEC-0023)**.
- **Estorno contábil da nota cancelada no Finance:** fora de escopo da SPEC-0016 (DL-0047); o
  cancelamento da NF preserva o documento arquivado (retenção legal). Entra no fluxo do Finance/Payout.
- **Telas Angular e métricas Micrometer (`invoices_issued_total`, `invoices_rejected_total`, latência
  do webservice):** backend-first; emissão/cancelamento/integração já são logados como evento de
  negócio com correlation id (follow-up para as métricas dedicadas, junto da SPEC-0027/UX).
