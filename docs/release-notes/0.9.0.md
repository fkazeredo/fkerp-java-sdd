# Release 0.9.0 — CommercialPolicy: parâmetros governados, precedência com proveniência e graduação do markup

- **Data:** 2026-06-29
- **Fase do ROADMAP:** 8a (CommercialPolicy — parâmetros governados)
- **Specs entregues:** SPEC-0014 (gradua o stub de markup da SPEC-0005)
- **Tag git:** `0.9.0`

## Destaques

A Fase 8a entrega o **motor de parâmetros governados** no módulo `com.fksoft.domain.commercialpolicy`
(o 12º Modulith, já existente — **graduado no lugar**, sem módulo novo): uma `ParameterRule` por
**camada** (`DIRECTIVE > PROMOTION > CONTRACT > POLICY > SYSTEM_DEFAULT`) × **escopo**
(global/conta/produto/canal) × **vigência**, e o motor `resolve(key, scope) → {value, provenance}`
que percorre a precedência e diz **qual camada venceu, quem definiu e quando** (redesenho 7.3). A
**diretiva do diretor** é o topo, com **auditoria reforçada** (papel + justificativa + evento), e
pode ser definida **em runtime** (self-service, Q8) — refletindo **imediatamente** na próxima
resolução. O ponto-chave: o **stub de markup da Fase 1 está graduado** — o `MarkupProvider` agora
resolve pelo motor e o markup que entra na cotação carrega a **camada vencedora** como `source` (não
mais o `SYSTEM_DEFAULT` fixo), **sem quebrar o Quoting** (contrato preservado; sem regra acima do
default a cotação ainda sai com `SYSTEM_DEFAULT`/markup 0 — back-compat). **239 testes verdes**, **12
módulos Modulith acíclicos**, 0 violações de Checkstyle; sem UI nesta fase (backend-first).

## Adicionado / Alterado

- **Added — Motor de precedência (SPEC-0014):** agregado `ParameterRule` (camada × escopo × vigência ×
  auditoria) em `…commercialpolicy.internal`; `ParameterResolver` **puro** com ordenação total
  determinística `(layer.rank, specificity DESC, validFrom DESC, createdAt DESC, id ASC)` (DL-0037);
  `CommercialPolicyService.resolve(key, scope)` retorna valor + `Provenance`; key sem `SYSTEM_DEFAULT`
  → **404 `policy.parameter.unknown`** (BR4). Value objects `ParameterKey`/`ParameterScope`/
  `ParameterValueType`/`ResolvedParameter`/`Provenance`. **Migração V18** com o seed dos `SYSTEM_DEFAULT`.
- **Added — Definição auditável de regras/diretivas em runtime (Q8, DL-0038):**
  `POST /api/commercial-policy/rules` (POLICY/PROMOTION/CONTRACT — papel admin/curador) e
  `POST …/directives` (camada DIRECTIVE — **papel diretor + justificativa obrigatória**, evento
  `DirectiveIssued`); sem papel → **403 `policy.directive.forbidden`**; valor/vigência malformados →
  **400 `policy.rule.invalid`**. `GET …/resolve` (Open-Host) e `GET …/rules` (auditoria/curadoria).
  Eventos `ParameterRuleDefined`/`DirectiveIssued` logados como evento de negócio.
- **Changed — Graduação do `MarkupProvider` (DL-0040):** o `CommercialPolicyService` **implementa** a
  porta; `currentMarkup(MarkupScope)` resolve `MARKUP_PCT` e reporta a **camada vencedora** em
  `source`; `currentMarkup()` delega para o escopo global (aditivo, retrocompatível). **Removido** o
  stub `SystemDefaultMarkupProvider`. `QuoteService.compose` passa o escopo da conta. `MarkupDecision`
  ganha as constantes de source das camadas, mantendo `SYSTEM_DEFAULT`.
- **Changed:** `HttpErrorMapping` (+3 códigos), i18n pt-BR + fallback en, `DevStubUserContextProvider`
  (+`ROLE_DIRECTOR`/`ROLE_POLICY_ADMIN`, stub até SPEC-0024), `OpenApiConfig` (descrição + versão),
  `pom.xml` → `0.9.0`.

## Contratos / API

- `GET /api/commercial-policy/resolve?key=&accountId=&productRef=&channel=` → `{key, value, type,
  provenance{layer, ruleId, definedBy, definedAt, validUntil}}` | 404 `policy.parameter.unknown`.
- `POST /api/commercial-policy/rules` → 201 (POLICY/PROMOTION/CONTRACT) | 403 | 400.
- `POST /api/commercial-policy/directives` → 201 (DIRECTIVE, justificativa obrigatória) | 403 | 400.
- `GET /api/commercial-policy/rules?key=&layer=` → lista para auditoria/curadoria.
- **Quoting consome via porta `MarkupProvider`, não via HTTP** — o JSON da Quote (markup.pct/source)
  é estável. Tudo **aditivo** (novo módulo de API/eventos; porta com sobrecarga retrocompatível) →
  **MINOR** (ADR 0015). OpenAPI via springdoc; erro estável `{code, message, fields}`.

## Decisões (decision-log)

- **DL-0037** — Modelagem `ParameterRule` (colunas de escopo, não jsonb) + motor de precedência com
  especificidade = nº de dimensões casadas e desempate determinístico — **Alta / Moderada**.
- **DL-0038** — **Q8 ASSUMIDO**: diretor/admin editam parâmetros e diretivas em **runtime**
  (self-service auditável; diretiva = papel diretor + justificativa + evento); **fluxos não** —
  **Média / Barata** (adota a Recomendação do ROADMAP).
- **DL-0039** — Seed `SYSTEM_DEFAULT` só das chaves **já usadas**; **Q5** (comissão do agente)
  **comportada** pelo motor por escopo, **sem** implementar Commissioning (segue aberta na SPEC-0004) —
  **Média / Barata**.
- **DL-0040** — Graduação do `MarkupProvider` **preservando o contrato** (sobrecarga com escopo;
  source = camada vencedora; sem regra → SYSTEM_DEFAULT) — **Alta / Moderada**.

> **decision-log/INDEX.md:** nenhuma DL desta fase é **Confiança=Baixa** ou **Reversibilidade=Cara**.
> DL-0037/DL-0040 são Alta/Moderada; DL-0038/DL-0039 são Média/Barata. As incógnitas de negócio
> (**Q5** — consumo da comissão do agente; conjunto final de chaves) **permanecem em aberto nas suas
> specs donas**, sem inventar regra (simulation-and-mocking.md).

## Riscos e pendências

- **Q5 — escopo de consumo da comissão do agente:** o motor a **comporta** (parâmetro governado por
  escopo, DL-0039), mas o **consumo é da SPEC-0004** — **em aberto lá**. Confirmar com o dono.
- **Conjunto final de `parameterKey`:** evolui por spec dona (SLA/ISS… entram com SPEC-0016/0018);
  o seed cobre só as chaves já usadas (não inventar default sem consumidor).
- **Papéis reais (`ROLE_DIRECTOR`/`ROLE_POLICY_ADMIN`):** hoje no `DevStubUserContextProvider`; o IdP
  real (papéis/OIDC) chega com **SPEC-0024** (Identity) — mesmo seam dos demais módulos.
- **Migração de Exchange/Reconciliation de constante → parâmetro governado:** Open-Host **expõe** o
  valor; o consumo fica para a fatia que escolher consultar o motor (não nesta).
- **Telas Angular / Micrometer:** backend-first nesta fase (follow-up).
