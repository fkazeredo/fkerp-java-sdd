# Release 0.26.0 — Fase 16c · Telas de operação: Inteligência/DSS, Política comercial, Marketing e Portfólio

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 16 (fatia 16c)
- **Specs entregues:** **SPEC-0029** (estendida — 16c implementa Intelligence/CommercialPolicy/Marketing/Portfolio)
- **Decisão de origem:** DL-0109 (gap de UI: telas de módulo adiadas nunca construídas)
- **Tag git:** `0.26.0`

## Destaques

A Fase 16c continua a **quitar a dívida de UI** (DL-0109), agora em **Inteligência & Crescimento**.
Entrega as **quatro telas de operação** da fatia — **frontend-only, sobre APIs que já existiam**
(nenhum endpoint novo, nenhum contrato ou schema tocado). Reusa o padrão estabelecido: service +
`API_BASE_URL` + `PageResponse`; `<app-screen-state>` para carregando/vazio/erro/permissão; rota lazy
sob o `Shell` com guards; nav por papel (só para esconder ruído de menu); i18n bilíngue; Vitest +
Playwright.

## Adicionado

- **Tela Inteligência** (`/intelligence`): lista os **insights** da DSS com filtros tipo/assunto/situação
  (ordenados por ganho estimado); abre um insight mostrando a **evidência** (subsídio acumulado, gap
  realizado, volume atraído e proveniência), a **recomendação** (veredito CONVERTE/QUEIMA_MARGEM, ação,
  ganho/risco) e o **guardrail** cruzado (alerta, nunca bloqueia); e **registra a decisão humana**
  (aceitar/rejeitar/descartar + nota). Registrar a decisão **só registra — nunca aciona ação** (a DSS
  aconselha; o humano decide). Consome `GET /api/intelligence/insights`, `GET /{id}`, `POST /{id}/decision`.
- **Tela Política comercial** (`/commercial-policy`): **resolve** um parâmetro governado por escopo
  (chave + conta/produto/canal) mostrando o **valor vencedor e a proveniência** (camada que venceu,
  quem/quando); lista as **regras** para auditoria exibindo a **precedência fixa** Diretiva > Promoção >
  Contrato > Política > Padrão; **define uma regra** POLICY/PROMOTION/CONTRACT (curador/diretor) e
  **emite uma diretiva** (topo da precedência; justificativa obrigatória; papel de diretor). Consome
  `GET /api/commercial-policy/resolve`, `GET/POST /rules`, `POST /directives`.
- **Tela Marketing** (`/marketing`): **consentimento LGPD** (estado atual + histórico append-only;
  conceder/revogar); **segmentos** (definir + prever alcance); **campanhas** (criar + disparar — o
  disparo filtra por consentimento e reporta alvos/suprimidos/enfileirados); **atribuição**
  campanha→reserva (registrar + listar); e **apagamento LGPD** (remove PII, mantém a tombstone de
  revogação). Consome os 11 endpoints de `/api/marketing`.
- **Tela Portfólio** (`/portfolio`): **marcas representadas** (listar/registrar/desativar); **contratos
  de representação** (listar + cobertura numa data — alerta, nunca bloqueia; registrar); e **metas ×
  realizado** (definir meta VOLUME/REVENUE + progresso meta × realizado × atingimento). Consome os 11
  endpoints de `/api/portfolio`.
- **Navegação e i18n:** quatro itens de nav novos (Inteligência/Marketing/Portfólio com
  `roles: ['ROLE_OPERATIONS']`; Política comercial com `roles: ['ROLE_DIRECTOR','ROLE_POLICY_ADMIN']`);
  blocos i18n pt-BR e en-US por tela. As telas cuja inicial colide com uma rota anterior seguem
  acessíveis pela paleta `Ctrl/Cmd+K`.
- **Testes:** Vitest por tela + specs de serviço (`HttpTestingController`) → **196 testes de frontend**,
  cobertura acima dos pisos da Fase 12; jornada Playwright `e2e/intelligence-policy.spec.ts` (diretor
  abre Inteligência → lista vazia → resolve parâmetro em Política comercial; token sem ROLE_DIRECTOR
  recebe 403 no endpoint de diretiva, o diretor é autorizado — autoridade no backend).

## Alterado

- **`backend/pom.xml` + OpenAPI**: versão `0.25.0 → 0.26.0` (apenas a string de versão e o texto da
  descrição do OpenAPI). `./mvnw verify` permanece **verde — 476 testes, 0 violações de Checkstyle**
  (nenhuma mudança de comportamento no backend).

## Não alterado (frontend-only)

- **Nenhum endpoint novo**, nenhuma migração, nenhum contrato/JSON/evento/schema mudado. As telas
  compõem os endpoints existentes (padrão do dashboard — DL-0094).

## Contratos / API

- **Sem mudança.** As telas consomem `/api/intelligence`, `/api/commercial-policy`, `/api/marketing` e
  `/api/portfolio` já existentes. A autorização é do backend: os GET/leituras e a maioria das ações
  exigem apenas autenticação; **as diretivas de política exigem `ROLE_DIRECTOR`** e as regras
  `ROLE_DIRECTOR`/`ROLE_POLICY_ADMIN` — a nav só reflete isso como tidiness.

## Como rodar / verificar

```bash
cd frontend && npm ci && npx ng lint && CI=true npx ng test --watch=false && npx ng build
# lint verde; 196 testes Vitest verdes; cobertura acima dos pisos (stmts 70.8% / branches 56.6% /
# funcs 50.1% / lines 76.1%); build gera os chunks intelligence/commercial-policy/marketing/portfolio-page.

cd backend && ./mvnw verify   # 476 testes, 0 violações de Checkstyle, BUILD SUCCESS (inalterado)

# E2E (stack isolado, Keycloak na 8089):
cd frontend && npm run e2e:up && E2E_BASE_URL=http://localhost:4201 npm run e2e && npm run e2e:down
# 17 specs Playwright (15 anteriores + a jornada Inteligência/Política comercial de 16c).
```

## Riscos e pendências

- **E2E no sandbox:** a jornada Playwright foi **autorada e compila** (o Playwright a descobre — 17
  testes via `npx playwright test --list`), mas **não foi executada aqui** porque a build da imagem do
  backend do stack isolado (`compose.e2e.yaml`) exige rede/cache Maven dentro do contêiner, indisponível
  no sandbox. O `./mvnw verify` **no host** está verde, então é limitação de infra, não defeito de código.
- **Dívida remanescente da Fase 16:** falta a fatia **16d** (People/RH, Ponto, Assets, Admin,
  Platform/TI, Identity/acesso → `0.27.0`).
