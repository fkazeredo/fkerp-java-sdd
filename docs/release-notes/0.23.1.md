# Release 0.23.1 — Upgrade de stack: Spring Boot 4.0.7 (manutenção interna, sem novidade ao usuário)

- **Data:** 2026-06-30
- **Fase do ROADMAP:** 14 (Upgrade de stack — **ADR + chore** de infra, sem SPEC)
- **Tipo:** **manutenção interna de infra**, **PATCH** (ADR 0015) — **sem mudança de contrato**
  (REST/DTO/JSON/evento publicado/i18n/schema) e **sem nova capacidade ao usuário** (Regra Zero)
- **Tag git:** `0.23.1`

## O que mudou (e o que NÃO mudou)

O backend foi migrado de **Spring Boot 3.5.16** para **Spring Boot 4.0.7** (Spring Framework 7, Jakarta
EE 11), com **Spring Modulith 2.0.7** e **springdoc-openapi 3.0.3**. Java permanece **21 (LTS)**;
Testcontainers **1.21.4**; Postgres **16-alpine**.

**Esta release não adiciona nenhuma funcionalidade visível ao usuário.** É um upgrade de plataforma:
nenhuma tela, navegação, endpoint, formato de erro, evento publicado ou dado mudou. **Por isso o MANUAL
do usuário (pt-BR e en-US) NÃO muda** — Regra Zero.

A serialização JSON da aplicação **continua idêntica**: a ponte `spring-boot-starter-classic` mantém o
classpath pré-4.0 (Jackson 2), então os 22 usos de Jackson 2 do código de produção (codecs de colunas
jsonb, `ObjectMapper` de adapters de integração/segurança) seguem inalterados. **O status HTTP 422
continua 422 na rede** — apenas a constante interna do framework foi renomeada
(`UNPROCESSABLE_ENTITY` → `UNPROCESSABLE_CONTENT`, alinhamento RFC 9110 do Spring Framework 7).

## Por que agora (e por que 4.0.x, não 4.1)

A `DL-0002` (Fase 0) fixou a stack em 3.5.16 por estabilidade, prevendo um ADR de upgrade "quando
houvesse motivo concreto". A Fase 14 **avaliou** e concluiu que o Boot 4.0.7 fica **verde** com correções
mecânicas, sem afrouxar nenhum portão — então **executou** o upgrade. Escolheu-se a linha **4.0.x
(4.0.7)**, a mais madura/testada, e **não a 4.1.0** (recém-GA), mantendo a disciplina de estabilidade.
**`@swimlane/ngx-graph` não foi trazido** (sem requisito real de workflow configurável — Regra Zero).

## Ajustes exigidos pelo Boot 4 (todos sem mudar comportamento)

- **Testcontainers BOM** importado explicitamente (o parent do Boot 4 não gere mais a versão).
- **`TestRestTemplate`** relocado (`org.springframework.boot.resttestclient`) e reativado por
  `@AutoConfigureTestRestTemplate` na classe-base de teste (cobre 41 testes de integração).
- **`@AutoConfigureObservability`** (removido no Boot 4) → **`@AutoConfigureMetrics`** nos 2 testes de
  métricas.
- **Leitura de `JsonNode`** nos testes de API migrada para Jackson 3 (`tools.jackson.databind.JsonNode`),
  pois o cliente HTTP de teste do Boot 4 desserializa com Jackson 3 (só o lado de teste).
- **Constante 422** atualizada para `UNPROCESSABLE_CONTENT` (3 testes + 1 mapeamento de produção).

## Sem migração, sem contrato

- **Nenhuma migração Flyway** (sem mudança de schema). **Nenhum** endpoint/DTO/JSON de fio/evento
  publicado/chave i18n alterado. OpenAPI: apenas a **versão** do documento sobe para **0.23.1**.

## Decisões (ADR + decision-log)

- **ADR 0017** — upgrade para Spring Boot 4.0 (Spring Framework 7, Spring Modulith 2.0); substitui a
  decisão de versão da DL-0002.
- **DL-0108** — sub-decisões: linha 4.0.7, ponte `spring-boot-starter-classic` (Jackson 2 mantido na
  produção, migração para Jackson 3 vira débito rastreado), e as correções de teste — Alta/Moderada.
- **DL-0002** — **superada na Fase 14** (nota de atualização apontando para ADR 0017 + DL-0108).

## Verificação

- **Backend** `./mvnw verify` **verde** (Spring Boot 4.0.7): **537 testes**, 0 falhas, 0 erros (mesmo
  número da 0.23.0 — nenhum teste apagado/pulado); JaCoCo INSTRUCTION ≈ **89,7 %** (piso 0,80);
  **Spotless**/**Checkstyle**/**ArchUnit**/**Spring Modulith** (22 módulos, acíclico) verdes.
- **Frontend** `lint`/`test`/`build` verdes (56 testes; cobertura Vitest/v8 OK) — independe do stack.

## Riscos / o que fica para depois

- A ponte `spring-boot-starter-classic` mantém **Jackson 2** no runtime — **débito rastreado em
  DL-0108**: migrar a produção para Jackson 3 (`tools.jackson`) e remover a ponte quando houver ganho
  concreto. Atenção: novos testes que leem corpo como árvore devem usar `tools.jackson.databind.JsonNode`.
- Adoção opcional de `RestTestClient` e avaliação de subir para Boot 4.1.x ficam para o futuro (DL-0108).
- **A Fase 14 era a última fase numerada do ROADMAP** (a 15 é regra+chore de documentação bilíngue, já
  vigente). Não há fase numerada seguinte.
