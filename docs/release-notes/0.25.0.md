# Release 0.25.0 — Fase 16b · Telas de operação: Pós-venda, Origem de ofertas, Mesa de câmbio e Cancelamento

- **Data:** 2026-07-01
- **Fase do ROADMAP:** 16 (fatia 16b)
- **Specs entregues:** **SPEC-0029** (estendida — 16b implementa AfterSales/Sourcing/Exchange-completo/Cancelamento)
- **Decisão de origem:** DL-0109 (gap de UI: telas de módulo adiadas nunca construídas)
- **Tag git:** `0.25.0`

## Destaques

A Fase 16b continua a **quitar a dívida de UI** (DL-0109), agora no **ciclo comercial**. Entrega as
**quatro telas de operação** da fatia — **frontend-only, sobre APIs que já existiam** (nenhum endpoint
novo, nenhum contrato ou schema tocado). Reusa o padrão estabelecido: service + `API_BASE_URL` +
`PageResponse`; `<app-screen-state>` para carregando/vazio/erro/permissão; rota lazy sob o `Shell` com
guards; nav por papel (`ROLE_OPERATIONS`, só para esconder ruído de menu); i18n bilíngue; Vitest +
Playwright. A tela de **Exchange** existente (taxa congelada) segue **intacta** — a mesa de câmbio é
uma tela **companheira**, não uma substituição.

## Adicionado

- **Tela Pós-venda** (`/aftersales`): lista chamados com filtros tipo/situação/reserva/**SLA estourado**;
  **abre** um chamado; conduz a **máquina de estados** (assumir/aguardar/encerrar) e **resolve** (aprovar
  reembolso → dispara Payout REFUND; aprovar cancelamento → dispara cancelamento na Booking; resolver sem
  ação; rejeitar). O `breached` é alerta ortogonal e **nunca bloqueia** (SPEC-0018 BR4). Consome
  `GET/POST /api/aftersales/cases`, `GET /{id}`, `POST /{id}/{assign|progress|wait|close}`,
  `POST /{id}/resolve`.
- **Tela Origem de ofertas** (`/sourcing`): **registra** a procedência de uma oferta (texto do produto,
  preço-base, origem, nível de integração, ref. externa) e **consulta** uma oferta por id. Consome
  `POST /api/sourcing/offers`, `GET /api/sourcing/offers/{id}`.
- **Tela Mesa de câmbio** (`/exchange-desk`): **exposição viva** do livro (subsídio acumulado + desvio
  mark-to-market + **alerta de drift**), **taxa de mercado** (registrar contingência manual + histórico),
  **posição por reserva** (subsídio × drift) e **relatório PromoFx** por período. Consome
  `GET /api/exchange/exposure`, `GET/POST /api/exchange/market-rates` (+`/current`),
  `GET /api/exchange/positions/{bookingId}`, `GET /api/exchange/reports/promo-fx`.
- **Tela Cancelamento** (`/cancellation`): **consulta e administra** a política por escopo
  produto/fornecedor — tipo, **janelas de penalidade**, reembolsável, quem arca, merchant of record,
  taxa de no-show; expõe a **armadilha do merchant** (ALL_SALES_FINAL ⇒ não reembolsável ao fornecedor).
  Consome `GET/PUT /api/products/{ref}/cancellation-policy`.
- **Navegação e i18n:** cinco itens de nav novos (Pós-venda/Origem de ofertas/Mesa de câmbio/Cancelamento
  com `roles: ['ROLE_OPERATIONS']`); blocos i18n pt-BR e en por tela. Corrige o mapa de atalhos `g`+tecla
  para manter a primeira tela quando duas rotas dividem a inicial (accounts/aftersales,
  exchange/exchange-desk, compliance/cancellation) — as novas seguem acessíveis pela paleta `Ctrl/Cmd+K`.
- **Testes:** Vitest por tela + specs de serviço (`HttpTestingController`) → **135 testes de frontend**,
  cobertura acima dos pisos da Fase 12; jornada Playwright `e2e/aftersales-cancellation.spec.ts` (usuário
  OPERATIONS abre Pós-venda → lista vazia → consulta política em Cancelamento; token sem ROLE_FINANCE
  recebe 403 no fechamento financeiro — autoridade no backend).

## Não alterado (frontend-only)

- **Nenhum endpoint novo**, nenhuma migração, nenhum contrato/JSON/evento/schema mudado. As telas
  compõem os endpoints existentes (padrão do dashboard — DL-0094). A tela de taxa congelada e seus
  testes seguem intactos.

## Contratos / API

- **Sem mudança.** As telas consomem `/api/aftersales`, `/api/sourcing`, `/api/exchange`
  (exposure/positions/market-rates/reports) e `/api/products/*/cancellation-policy` já existentes. A
  autorização é do backend; hoje estes GET/POST exigem apenas autenticação (sem matcher de papel
  específico) — a nav marca `ROLE_OPERATIONS` só como tidiness.

## Como rodar / verificar

```bash
cd frontend && npm ci && npx ng lint && CI=true npx ng test --watch=false && npx ng build
# lint verde; 135 testes Vitest verdes; cobertura acima dos pisos (stmts 70.8% / branches 58.1% /
# funcs 49.5% / lines 74.7%); build gera os chunks aftersales/sourcing/exchange-desk/cancellation-page.

cd backend && ./mvnw verify   # 476 testes, 0 violações de Checkstyle, BUILD SUCCESS (inalterado)

# E2E (stack isolado, Keycloak na 8089):
cd frontend && npm run e2e:up && E2E_BASE_URL=http://localhost:4201 npm run e2e && npm run e2e:down
# 15 specs Playwright (13 anteriores + a jornada Pós-venda/Cancelamento de 16b).
```

## Riscos e pendências

- **E2E no sandbox:** a jornada Playwright foi **autorada e compila** (o Playwright a descobre — 15
  testes via `npx playwright test --list`), mas **não foi executada aqui** porque a build da imagem do
  backend do stack isolado (`compose.e2e.yaml`) exige rede/cache Maven dentro do contêiner, indisponível
  no sandbox. O `./mvnw verify` **no host** está verde, então é limitação de infra, não defeito de código.
- **Dívida remanescente da Fase 16:** faltam as fatias **16c** (Inteligência/DSS, CommercialPolicy,
  Marketing, Portfolio → `0.26.0`) e **16d** (People/Ponto, Assets, Admin, Platform/TI, Identity →
  `0.27.0`).
