# Release 0.24.0 — Fase 16a · Telas de operação: Financeiro, Faturamento, Repasses e Conformidade

- **Data:** 2026-06-30
- **Fase do ROADMAP:** 16 (fatia 16a)
- **Specs entregues:** **SPEC-0029** (nova — telas de operação; 16a implementa Finance/Billing/Payout/Compliance)
- **Decisão de origem:** DL-0109 (gap de UI: telas de módulo adiadas nunca construídas)
- **Tag git:** `0.24.0`

## Destaques

A Fase 16a começa a **quitar a dívida de UI** diagnosticada em **DL-0109**: o backend tem 22 módulos e
37 controllers REST, mas o frontend só tinha tela para ~5. Esta fatia **autora a SPEC-0029** e entrega
as **quatro primeiras telas de operação** — **frontend-only, sobre APIs que já existiam** (nenhum
endpoint novo, nenhum contrato ou schema tocado). Reusa o padrão estabelecido: service +
`API_BASE_URL` + `PageResponse`; `<app-screen-state>` para carregando/vazio/erro/permissão; rota lazy
sob o `Shell` com guards; nav por papel; i18n bilíngue; Vitest + Playwright.

## Adicionado

- **Tela Financeiro** (`/finance`): razão AP/AR com filtros direção/situação/período/parte; formulário
  de novo lançamento; consulta de período com **balancete por moeda** (DL-0013 — nunca soma moedas
  diferentes) e **fechamento do mês** (mostra `finance.period.cannot-close` quando o Compliance veta).
  Consome `GET/POST /api/finance/entries`, `GET /api/finance/periods/{yyyymm}`, `.../trial-balance`,
  `POST .../close`.
- **Tela Faturamento** (`/billing`): consulta a NF de comissão por id (base/ISS/retenções/situação/
  número), **cria rascunho**, **emite** (exige ROLE_FINANCE) e **cancela**. Consome `GET/POST
  /api/billing/invoices`, `POST .../issue`, `POST .../cancel`.
- **Tela Repasses** (`/payouts`): lista repasses/liquidações/reembolsos por tipo/situação/beneficiário,
  abre um com suas parcelas, **cria** e **executa** (falha do provedor vira **FAILED** explícito, nunca
  um falso EXECUTED). Consome `GET/POST /api/payouts`, `GET /{id}`, `POST /{id}/execute`.
- **Tela Conformidade** (`/compliance`): roda a **verificação de fechamento** (pode-fechar / pendências
  e o que falta), **envia** documento ao cofre e **consulta** documento por id (tipo/hash/emissão/
  retenção/dado pessoal; o `fileRef` interno nunca é exposto). Consome `GET /api/compliance/close-check`,
  `GET/POST /api/compliance/documents`.
- **Navegação e i18n:** quatro itens de nav (Financeiro/Faturamento/Repasses com `roles:
  ['ROLE_FINANCE']`; Conformidade visível a qualquer autenticado — o backend continua a autoridade, isto
  só esconde ruído de menu); blocos i18n pt-BR e en por tela.
- **Testes:** um Vitest por tela (89 testes de frontend, cobertura acima dos pisos da Fase 12) e uma
  jornada Playwright de Financeiro/Conformidade (`e2e/finance.spec.ts`): usuário FINANCE abre Financeiro
  → razão vazio → close-check em Conformidade; token não-FINANCE recebe 403 no fechamento e um token
  FINANCE passa o portão.

## Alterado

- **`backend/pom.xml` + OpenAPI**: versão `0.23.1 → 0.24.0` (apenas a string de versão). `./mvnw verify`
  permanece **verde — 476 testes, 0 violações de Checkstyle** (nenhuma mudança de comportamento no
  backend).

## Não alterado (frontend-only)

- **Nenhum endpoint novo**, nenhuma migração, nenhum contrato/JSON/evento/schema mudado. As telas
  compõem os endpoints existentes (padrão do dashboard — DL-0094).

## Contratos / API

- **Sem mudança.** As telas consomem `/api/finance`, `/api/billing`, `/api/payouts` e
  `/api/compliance` já existentes. Ações sensíveis seguem exigindo o papel (fechar mês / emitir NF →
  ROLE_FINANCE) → 403 auditado; sem token → 401 genérico.

## Como rodar / verificar

```bash
cd frontend && npm ci && npx ng lint && CI=true npx ng test --watch=false && npx ng build
# lint verde; 89 testes Vitest verdes; cobertura acima dos pisos (stmts 76.9% / branches 59.4% /
# funcs 50.1% / lines 78.7%); build gera os chunks finance-page/billing-page/payout-page/compliance-page.

cd backend && ./mvnw verify   # 476 testes, 0 violações de Checkstyle, BUILD SUCCESS (inalterado)

# E2E (stack isolado, Keycloak na 8089):
cd frontend && npm run e2e:up && E2E_BASE_URL=http://localhost:4201 npm run e2e && npm run e2e:down
# 13 specs Playwright (11 anteriores + a jornada Financeiro/Conformidade de 16a).
```

## Riscos e pendências

- **E2E no sandbox:** a jornada Playwright foi **autorada e compila** (o Playwright a descobre — 13
  testes), mas **não foi executada aqui** porque a build da imagem do backend do stack isolado
  (`compose.e2e.yaml`) falhou **dentro do contêiner** por `DependencyResolutionException` do Maven (sem
  `.m2` no build). O `./mvnw verify` **no host** está verde (as dependências resolvem), então é uma
  limitação de infra do sandbox, não defeito de código. Rodar num ambiente com rede/cache Maven para o
  build da imagem executa a jornada.
- **Dívida remanescente da Fase 16:** faltam as fatias **16b** (ciclo comercial — AfterSales, Sourcing,
  Exchange-completo, Cancelamento → `0.25.0`), **16c** (Inteligência/DSS, CommercialPolicy, Marketing,
  Portfolio → `0.26.0`) e **16d** (People/Ponto, Assets, Admin, Platform/TI, Identity → `0.27.0`).
