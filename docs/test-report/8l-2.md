# Caderno de Testes — Fatia 8l-2 (Admin: despesa → Finance + documentos exigidos)

> Spec: **SPEC-0025** (Admin). Migração **V30** (catálogo do Compliance estendido com o `SERVICE`).
> Decisões: DL-0085 (mapa `kind`→`EntryType`→documento), DL-0086 (integração por fachada/porta,
> idempotência). Release alvo **0.20.0**.

## Escopo

Registrar uma **despesa recorrente** cria o lançamento certo no Finance (via fachada
`FinanceService.register`, sem FK), guarda o `financeEntryId` por valor e **aponta os documentos
exigidos** pelo Compliance (porta de leitura `DocumentRequirementDirectory`). Cobre os **Acceptance
Criteria**:

- Despesa de energia (UTILITY) → lançamento PAYABLE/UTILITY_EXPENSE + `requiredDocuments=[UTILITY_BILL]`.
- Mapa por tipo: SERVICE→SERVICE/[NFSE]; autônomo→AUTONOMOUS_SERVICE/[RPA]; OTHER→OTHER_EXPENSE/[].
- Idempotência por `(supplier, period, kind)` — duplicada → 409 `admin.expense.duplicate`, sem 2º lançamento.
- **Regra de ouro (regressão):** despesa **sem** documento bloqueia o fechamento do Finance via Compliance.

## Resultado

`cd backend && ./mvnw verify` → **BUILD SUCCESS** (medido no fim da fase). Esta fatia adiciona 5
casos de integração. Migração V30: catálogo `document_requirements` ganhou `SERVICE`/`NFSE`
(AT_REGISTRATION) + `SERVICE`/`PAYMENT_PROOF` (AT_SETTLEMENT), sem editar a V8.

## Cobertura — o que NÃO está coberto (e por quê)

- **Liquidação/baixa da despesa** (PAYMENT_PROOF na fase AT_SETTLEMENT): o pagamento em si é Payout
  (SPEC-0017); aqui só o registro + o requisito no fechamento. O seed AT_SETTLEMENT já existe.
- **Confirmação do lançamento** (PROVISIONAL→CONFIRMED): é operação do Finance (SPEC-0015), fora do
  escopo do Admin (BR4 — Admin só gera o lançamento).

## Como reproduzir

```bash
cd backend
./mvnw -o test -Dtest='AdminExpenseIntegrationTest'   # Docker no ar
./mvnw verify
```
